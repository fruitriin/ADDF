# モデル配分ガイド（マルチモデル運用）

役割ごとに異なる Claude モデルを使い分ける運用の考え方。手順・エージェント定義の詳細は
[`.claude/agents/addf-implementer.md`](../../agents/addf-implementer.md) や各エージェント定義本文が正。
このガイドは配分の考え方と3つの実現手段の概観。

## なぜ配分するか

モデルにはそれぞれ異なる特性（推論バジェットの深さ・意図の汲み取り・計画遵守能力・
美的感覚・応答速度）がある。ADDF のタスクは性質が大きく異なる複数の役割
（判断・実装・レビュー・文言調整）に分かれるため、役割ごとに得意なモデルを充てることで
全体の速度と質を両方上げられる。

## 配分の考え方（特徴ベース、固定式にしない）

投機適性の3区分（[speculative-development.md](speculative-development.md)「投機適性」節）や
7割共有原則（CLAUDE.md「迷ったときの作法」）と同じく、モデル選定も**特徴ベースの判定**であり、
数値スコアの固定式にはしない。目安となる能力軸:

| 能力軸 | 効いてくる役割 |
|---|---|
| 推論の深さ・粘り強さ | 難所の実装、複雑なデバッグ、Plan 起草 |
| 意図の汲み取り・コモンセンス | オーナー対話、判断の要る運転手役 |
| 計画遵守・やり遂げる力 | Plan/スコープが明示された実装 |
| 「自分の仕事を疑う」メタ認知 | レビュー（skeptic/attacker 系ペルソナ） |
| 美的感覚・初見の読みやすさ | UI/文言、newcomer ペルソナ、ドキュメント |
| 応答速度 | 頻度の高い定型判断（ループの運転手） |

これらの軸に沿って、プロジェクトごとに使えるモデルの特性を評価し、役割に割り当てる。
評価は `CLAUDE.repo.md` の「モデル配分ポリシー」節（`CLAUDE.repo.example.md` にプレースホルダあり）
に書き残すと、次にセッションを引き継ぐエージェント（別モデルの可能性がある）が読んで従える。

### 参考実例（出典: ダウンストリーム wasurenainder、2026-07-07 オーナー起案）

```
メインループ運転手（判断・選択・検収・対話）        → 推論バジェット小・高速・意図汲み取り重視のモデル
実装サブエージェント（Plan/スコープに従う実装）      → 計画遵守・やり遂げる力重視のモデル
レビュー（通常 + skeptic/attacker ペルソナ）        → 「自分の仕事を疑う」メタ認知重視のモデル
レビュー（newcomer ペルソナ）・UI/文言・ドキュメント → 美的感覚・初見の読みやすさ重視のモデル
Plan 起草・難所                                    → 最上位モデル（予算が許す範囲。エスカレーション参照）
```

これは一例であり、実際の表はプロジェクトが使えるモデルの組み合わせと評価によって変わる。

## 実現手段（3つ）

1. **エージェント定義の `model:` frontmatter**（`.claude/agents/*.md`）— エージェント単位で固定する。
   最も細かい粒度。`addf-implementer` のようにロール専任のエージェント定義を作り、そこに
   割り当てるのが基本形
2. **メインセッションの `/model` コマンド** — ループ全体（運転手役）の既定モデルを切り替える。
   セッション開始時や、モデルの予算状況に応じて手動で切り替える運用が基本
3. **`Agent` tool 呼び出し時の `model` パラメータ** — 1回限りのオーバーライド。エージェント定義の
   既定を上書きしたいとき（普段は Sonnet 系エージェントだが今回だけ難所なので上位モデルで、等）に使う

### tier エイリアス vs フル ID ピン留め

`model:` の値には tier エイリアス（`opus`/`sonnet`/`haiku` 等）とフルモデル ID の両方を書ける。

- **エイリアス**（デフォルト推奨）: その時点の最新版に自動追従する。配布物（ADDF 本体の
  エージェント定義）はモデル世代に依存しないためエイリアスを使う
- **フル ID でのピン留め**: ダウンストリームが特定世代に既知の問題（ツール呼び出しの不安定さ等）
  を把握している場合に使う一時的な回避策。ピン留めした理由をコメントで残し、新版で問題が
  解消されたら見直す運用にする（放置すると古い世代に固定されたまま気づかれなくなる）

## エスカレーション基準

実装が難航したとき、無理に同じモデル・同じ試行を繰り返さず、次のいずれかへ逃す:

- **同一サブタスクで実装が2回失敗した**、または**着手前の見積もりでステップ数が大きいと分かった**
  → 実装を止め、調査だけを委譲して Plan を厚くしてから再実装する
- 判断の要素が強く、実装側モデルの守備範囲を超える → `.claude/addf/Questions.md` に置いて
  オーナー判断を仰ぐ（7割共有原則の既存フローと同じ）
- 今すぐ着手する必要はない → TODO に「高性能モデル待ち」等のラベルを付けて一旦退避する
  （高価なモデルの予算が回復してからまとめて着手する運用も想定）

## 7割共有原則の補正（推論バジェットが小さいモデルで運転する場合）

推論バジェットが小さく応答速度優先のモデルでメインループ（`/addf-dev` の運転手役）を回す場合、
CLAUDE.md「迷ったときの作法（7割共有原則）」の閾値判定に以下を補正として加える:

- **自前で長い実装をしない**: 判断は運転手が行い、実装は必ずサブエージェント（`addf-implementer`
  等）へ委譲する。ステップ数の多い作業を自力で進めて途中で切り上げる（「できない」と諦める）
  失敗モードを避けるため
- **閾値は保守側（`nervous` 寄り）に倒す**: 確信度の見積もり自体の精度がモデルによって変わりうる
  ため、疑わしいときは早めに Questions.md へ置く運用にする
