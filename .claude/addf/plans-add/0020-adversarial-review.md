## Context

ADDF の品質ゲートでは `addf-code-review-agent` と `addf-security-review-agent` がコードをレビューする。
しかしこれらのレビュアーは実装者と同じモデルを使うため、**前提を共有してしまい盲点も共有しがち**。

Fable からの一次フィードバック:

> code-review や security-review のサブエージェント、中身は私と同じモデルだから、盲点も共有しがち。
> 「実装者の前提を疑え」「壊す目的で読め」みたいに敵対的な役割を強めに与えると、セルフレビューの甘さが減ると思う。

オーナー指針:

> レビューのときの盲点を見つけるための違う視点を置いたレビューの設計。

## 設計

### 1. 視点（ペルソナ）の定義

レビュー時にエージェントへ与えるロールを複数定義する:

| ペルソナ | 役割 | プロンプト要旨 |
|---|---|---|
| `skeptic` | 懐疑論者 | 「実装者の前提を全て疑え。何を当たり前と思って書いたか問え」 |
| `attacker` | 攻撃者 | 「このコードを壊す目的で読め。境界・想定外入力・並行アクセス・型の隙間を狙え」 |
| `newcomer` | 新人視点 | 「このコードを今初めて読む。命名・構造で意図が分からない箇所を全て指摘せよ」 |
| `maintainer` | 半年後の保守者 | 「半年後にバグ修正で読む。依存関係・暗黙の前提・テストの抜けを指摘せよ」 |
| `domain-skeptic` | ドメイン懐疑 | 「ドメイン要件と実装の乖離を疑え。Plan と実装が同じものを指しているか検証せよ」 |

### 2. `addf-code-review-agent` の役割割当

既存のレビューエージェントを、ペルソナを引数で受け取れる構造に拡張する。

各エージェントは独立したコンテキストで、自分のペルソナのみを念頭にレビューする。
結果はメインエージェントが集約して重複排除・重要度判定する。

```
addf-code-review-agent --persona=skeptic
addf-code-review-agent --persona=attacker
addf-code-review-agent --persona=newcomer
addf-code-review-agent --persona=maintainer
```

### 2.5. 常設しない — 起動タイミングの限定

オーナー指針:

> 常設で起動すると重そうなので、自律感強いときとかマイルストーンのタイミングに呼び出すの限定にするとかのほうがいいかもねえ。

**通常タスクでは現行の単一レビューを維持**し、複数ペルソナ並列起動は限定的に発動する。

| タイミング | レビュー体制 |
|---|---|
| 通常タスク完了時 | `addf-code-review-agent` 単体（現行） |
| `/loop` 長時間自走中の Phase 完了 | 単体（現行）。ただし `mode: unattended` 時は3体並列 |
| マイルストーン（Phase 群の完了、リリース直前） | **3〜5体並列** |
| Plan の `mode: critical` 宣言時 | **5体並列** |
| オーナーが `/addf-review --persona-panel` で明示指示 | **3〜5体並列** |

「自律感が強い」状況（unattended モード、長時間 loop）はエージェント単独で品質を担保する必要があるため、
**ユーザー監督が弱まるほどレビューを厚くする**運用にする。これは Plan 0016 の「ユーザー応答性軸」と整合する。

通常のインタラクティブな作業ではオーナーが目視で確認するので、ペルソナ並列はコスト過多になる。

### 3. `addf-security-review-agent` の敵対性強化

セキュリティレビューは元々「攻撃者視点」が本質なので、プロンプトを強化:

```
あなたはペネトレーションテスターです。このコードを「壊す・侵入する・データを盗む」ために読みます。
「正しい使い方をされた場合」は無視し、「悪意ある入力・誤操作・並行アクセス・タイミング攻撃」を専門に探します。
善意の解釈を一切しないでください。
```

### 4. 集約ロジック

複数ペルソナの指摘は重複しやすい。メインエージェントの集約手順:

1. 各エージェントの指摘を Critical/High/Medium/Low/Info で分類
2. 同一箇所・同一原因の指摘は重複として1件にまとめ、ペルソナを列挙
3. **2ペルソナ以上が独立に指摘した項目**は重要度を1段上げる（コンセンサス補正）
4. 1ペルソナのみの指摘は元の重要度のまま（独自視点として尊重）

### 5. 既存スキルとの整合

- `code-review` スキル（プラグイン由来）はそのまま残し、ADDF 固有のペルソナ分割は `addf-code-review-agent` のみに適用
- `/code-review ultra` を呼ぶ場合は既存挙動を尊重

### 6. オプション: 「judge panel」モード

Plan が複雑な場合、ペルソナを3体並列実行 → 結果を別エージェント `addf-review-synthesizer` が判定するモードも導入できる。
本 Plan では `addf-code-review-agent` 内の集約で済ませ、synthesizer は将来 Plan で検討。

## メモ

- ペルソナ並列は常設しない（節 2.5）。マイルストーン or 自律走行時のみ発動
- ペルソナを増やすほど並列コストが上がる。マイルストーン時のデフォルトは `skeptic + attacker + newcomer` の3体、`mode: critical` 宣言時は5体すべて起動
- 同じモデルでもプロンプトのロールで盲点はずらせる（一部）が、根本的な盲点解消には「複数モデル」の利用も将来検討
- セキュリティレビューは ADDF 本体ではダウンストリームに必須化しない（プロジェクトの性質次第）
