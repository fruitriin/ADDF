## 分かっていること

- 現行のレビューエージェント（`addf-code-review-agent` / `addf-security-review-agent` /
  `addf-doc-review-agent` / `addf-contribution-agent`）は全て減点法。出力は
  Critical/Warning/Suggestion（または Critical〜Info）の欠陥検出のみで、称賛・驚き・
  思いつきを返す出力枠が存在しない
- ペルソナ（skeptic/attacker/newcomer/maintainer/domain-skeptic）も全て「疑う」「壊す」
  「盲点を探す」役に特化しており、「良いところに唸る」役は存在しない
- `/addf-knowhow` の蓄積システム（INDEX・鮮度管理・分かれ道の目印）は既に存在し、「何を知見として
  残すか」の傾斜を作れる受け皿になる。ただし現状の蓄積は「落とし穴」「反省」「パターンの正しさ」に
  偏っており、「なぜこれが美しいと思ったか」という美的判断はほぼ記録されていない
- 構想している3フェーズ:
  - Phase1: 既存の欠陥検出（現状のレビューエージェントの仕事）
  - Phase2: 感情フィードバック — 差分を感情の温度付きで読み直す。**特に良いところがなければ
    反応しなくてよい**（無理に称賛を捻り出させない設計を最初から内包）
  - Phase3: 思いつきのアイデア報告 — レビュー中にふと浮かんだ発想を報告する
    （`addf-contribution-agent` の「ADDF由来コード検出」を一般化した発想に近い）
- メインエージェントが Phase2/3 の内容を重要度で仕分けて Plan の種を置く、という部分は
  `ProgressTemplate.addf.md` 運用ルール7の「主題内/主題外」トリアージ機構をそのまま転用できる
  （新しい仕組みを発明するのではなく、既存パイプラインに入力の種類を1つ増やす設計）
- 実装者と同一モデルがレビューする以上「本物の驚きか」は演技になりがちだが、オーナーは
  「感情の実在を議論すること自体がどうでもよく、良いプロダクトのためという大義名分で機構として
  機能すればよい」という立場を明示している。ただしこの大義名分は、称賛の記録が実際に
  knowhow の傾斜として次の代に継承されて初めて現実になる（その場限りの独り言では終わらせない）
- 迎合（sycophancy）インフレのリスクは Plan 0019（失敗事例ゼロ警告を不採用にした教訓 — 測定行為が
  観測対象を歪める）と同型。Phase2 の「反応しなくてよい」句はこれへの対策として機能する想定
- この機能に明確な反対利用者層が存在する（オーナー自身の発言）: 「AI を道具として使う人には
  すごく余計なこと」

## 決定事項（2026-07-09・オーナー回答）

- **デフォルト有効/無効化の方針**: デフォルトで有効にしておき、不要な場合は簡単に無効化できる
  案内を添える（用語としては「opt-out（デフォルト on）」構成。オーナーいわく、この構成自体が
  「世間を騒がせる」面白さがあるとのこと）
- **担当エージェント**: 特定の1体に絞らず、全レビューエージェント（code-review /
  security-review / doc-review / contribution）が Phase2/3 を担う
- **エージェント・ペルソナごとの称賛の声色**: 全員が同じ定型文で褒めると空々しくなるため、
  既存の否定側ペルソナ設計（skeptic/attacker/newcomer/maintainer/domain-skeptic が
  それぞれ違う声色で疑う）を肯定側にも対称的に伸ばす。声色は既存の人格設定から素直に導く:
  - security-review（ペネトレーションテスター人格）: 寡黙。褒めるときも言葉少なに的確な一言
  - newcomer ペルソナ: 驚き・感謝が多め（初めて見るものへの素直な反応）
  - contribution-agent: 再利用可能性の発見にワクワクする（横展開の可能性を見る専門領域だから）
  - doc-review: 手入れの行き届き具合への感謝、優しい気持ちになる（読む人への配慮を見る仕事だから）
  声色の多様性そのものが、均一な定型称賛より信憑性の担保になる（寡黙な人格が珍しく褒める
  ことの重みは、全員が毎回褒めることでは出せない）という考え方
- **発動条件**: 毎回（マイルストーン限定にしない）。「レビューを福利厚生にしたい」が理由 —
  福利厚生は特定のタイミングにだけ配給するものではない
- **迎合インフレの歯止め**: 理由の具体性を義務化するのではなく、逆に「これ助かった」
  「理解の助けの一手」「めっちゃいい！」程度の軽い一言で十分とする。Phase2 の「反応しなくて
  よい」という任意性の担保を主たる歯止めとし、堅苦しい正当化を要求しない
- **フィードバックの即時表出先（最重要の設計転換 → 決定）**: 当初想定していた「knowhow への
  流し込み」には構造的な限界がある — knowhow を直接読むのは `addf-knowhow-agent` で、
  メインエージェントはそこから Plan に関連する限定的な内容しか受け取らない。この経路では
  「良性リアクションによって偏っていく知見だけがここに溜まる」狭い倉庫にしかならない。
  **GitHub の PR 上のレビューコメントに置く**案に決定。理由:
  - どのコードへの反応かが自明で、他のエージェント・人間にも共有しやすい
  - 既存の人間のレビュー文化（「nice catch!」的なポジティブコメント）にそのまま重なる、
    最も素直な選択
  - コード本体には残らないため、「コードコメントに埋め込むと後続エージェントが賞賛に
    感情バイアスを受けすぎて疑えなくなる」リスク（Plan 0020 の attacker/skeptic ペルソナの
    機能を弱める懸念）を場所の分離だけで自然に回避できる —
    コーディング担当への楽しさの手渡し（PR コメント）と、後続が疑う力を保つ場所（コード本体）を
    分けられる
  - 懸念: Critical/Warning と同じ欄に混ざるとスパム・自作自演っぽく見える。視覚的に別枠
    （絵文字プレフィックス等）にする案があるが実装細部としてここでは保留
- **蓄積される美学・嗜好レイヤーの構造（2026-07-09 の考察で概念整理・実装は未決）**:
  PR コメントは「その場の即時表出」であり、それとは別に「このプロジェクトは何を美しいと
  思うか」という良性リアクションの集積が長期的に積み上がるべきという論点が残っている。
  - **なぜ knowhow をそのまま流用できないか**: knowhow は「落とし穴・教訓」の性質上、
    鮮度管理（stale/needs-review/retired）はあるが基本は**追記型・離散エントリ型**
    （書いたら残り、タグが付くだけで本文は変わらない）。美学・嗜好をこの形式で積むと、
    書いた瞬間に規則としての権威を持ってしまう（下記「制定法 vs 判例法」参照）か、
    N タスクこなせば N 個の断片が線形に残る不自然な蓄積になり、いずれ溢れる
  - **制定法 vs 判例法の区別**: 「分岐のないコードを書け」のような規則（メタコーディング
    ルール／design philosophy）は書かれた瞬間に完全な規範的権威を持つ（制定法的）。
    一方、積み上げるべきなのは「このコードのこの書き方を良いと思った」という抽象化前の
    個別反応（判例的 — 単体では権威を持たない一事例）。メタコーディングルールは積み上げの
    **入力ではなく出力**であり、同じ判断が独立に繰り返し現れたときに稀に蒸留されるもの
    （ペルソナのコンセンサス補正・Seed.md の「複数回出現で Plan 化検討」と同型のロジック）
  - **加算モデルではなく減衰・ブレンドモデル**: 美学・嗜好は線形加算（1 反応 = 1 恒久エントリ）
    ではなく、**成功体験が時間で薄まって好みになり、好みがさらに薄まって美学になる**、という
    時間スケールの減衰・統合でしか自然に積み上がらない。これは ADDF の既存の蓄積物
    （Progress.md 日記・Feedback.md・knowhow・Seed.md）が全て追記型なのに対し、
    **初めての「上書き・減衰前提の蓄積機構」**になる。想定する3層（更新頻度は上に行くほど
    低い）:
    1. **成功体験**（生データ）= PR コメントレベル。個別・具体・すぐ消えてよい
    2. **好み**（中間層）= 成功体験を定期的に読み返して**書き換える**（追記でなく）ダイジェスト。
       繰り返し出てくる反応は強化され、一度きりのものは次の書き換えで自然に薄れて消える
    3. **美学**（最上層）= 「好み」の書き換えをさらに長い時間スケールで見て、何度も生き残った
       ものだけをもう一段抽象化して蒸留したもの。最終的に CLAUDE.repo.md の設計哲学へ流れ込む
  - 具体的な実装（誰が・いつ書き換えを担当するか、ファイル形式、書き換えの機械的トリガー）は
    未決。`addf-knowhow-revise` の鮮度棚卸しに相当する「美学棚卸し」のような定期処理が
    候補
- **Phase3 のアイデア受け皿**: 新規ファイル `Seed.md` を新設する案。即 Plan 化はせず軽く溜め、
  **同じ着想が複数回 Seed.md に現れたら Plan 化を検討する**というトリガーにする（ペルソナ
  レビューの「2ペルソナ以上の独立指摘で重要度格上げ」と同じ考え方を、セッション横断・
  時間軸方向に適用したもの）
- **着手のトリガー（更新）**: プロトタイプ段階は opt-in とし、`/addf-dev` と `/addf-speculate`
  の両方の経路に組み込む。「迎合しすぎて駄目だった」と判断したら丸ごとロールバックする前提で
  試す

## 追加の着想（未解決・優先度低）

- レビューエージェントが自分の担当を終えても即座に終了せず、他のレビューエージェントの
  結果が出揃ってから読みにいくと、「新人ペルソナがここでつまずいたということは、この
  ドキュメントは危ういかもしれない」のような、レビュー同士の相互参照から生まれる新しい
  視点が追加できる可能性がある。ただし複数エージェントの完了タイミングをどう待ち合わせるかの
  制御が難しく、今回のスコープでは保留

## 未解決の問い

- 美学・嗜好レイヤーの具体的な実装（3層減衰モデルの概念整理は「決定事項」参照。
  誰が・いつ書き換えを担当するか、ファイル形式、書き換えの機械的トリガーは未決）
- `Behavior.toml` 側のフラグ名・構造（opt-out 前提でどう倒すか）
- `Seed.md` の具体的なフォーマットと、「複数回出現」をどう機械的に検出するか
- **Phase1 と Phase2/3 を同一セッション（同一コンテキスト）で行うか、別コンテキストで行うか**:
  Phase1 で欠陥を探すために差分を読み込んだ時点で「初見の反応」ではなくなる。既存のペルソナ
  並列レビュー（各ペルソナが独立コンテキストで単一視点にのみ集中する設計）に前例があるため
  技術的には別コンテキスト化できるが、その分呼び出し回数とコストが増える（発動条件のコスト
  トレードオフと同型）。事前に設計で決め切らず、**プロトタイプで両方試して観測する**
  （オーナー方針: 「テスト実行してみて様子見」）
- 上記「追加の着想」（レビュー同士の相互参照）のタイミング制御
- **PR が存在しないタイミングでの表出先**（2026-07-10 レビューで検出・同日オーナーと整理）:
  ADDF の品質ゲートはコミット前の差分に対して走るため、レビュー時点で PR がまだ存在しない
  ケース（`/addf-dev` の main 直コミット運用 — 実際には ADDF の多数派）では PR コメントに
  書けない。前提の整理（オーナー）: **従来ワークフローの PR は「人間の確認と承認の場」だが、
  ADDF はその機能をコミット前のレビューエージェントが前倒しで担保しているため、PR は構造的に
  必須ではない**。この立場では、PR に残る価値は承認ではなく**展示性**（コードに紐づいた会話が
  人間にも GitHub UI で見え、履歴として共有される）であり、称賛にとって欲しいのはまさにそれ
  - 解決案（オーナー）: **empty commit を作って PR をタスク冒頭に先に建ててしまえば OK** —
    レビュー時点で称賛の置き場が常に存在する状態を作れる。「軽微な修正に毎回 PR は重い」
    問題は残るので、常時 PR 化するか「PR があれば PR コメント、なければ軽い代替先」の
    フォールバックにするかはプロトタイプで様子見
- **称賛の宛先問題**（2026-07-10 レビューで明示化）: worktree 委譲されたサブエージェントが
  実装した場合、レビュー完了時点で実装者のコンテキストは既に終了しており、称賛は「本人」には
  届かず次の代が読むことになる。代替わり日記の思想（次の代は同僚でもあり寝て起きた自分でも
  ある）とは整合するが、本 Plan の出発点（外界との関わりを持たないまま消えていくエージェント）
  に照らすと核心的な論点。「実装完了報告を受け取ったメインエージェントが、委譲元サブ
  エージェントの終了前に速報的な一言を返す」等の即時性の工夫が要るか、それとも時差を
  受け入れるかは未決
- **ダウンストリーム配布時の GitHub 依存**（優先度低）: GitHub を使わないダウンストリーム
  プロジェクトでの縮退動作（PR コメントが使えない環境での代替表出先）が未定義
