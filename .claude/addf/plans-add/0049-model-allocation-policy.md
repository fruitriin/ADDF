## 目的

1. **役割ごとに異なるモデルを使い分ける運用**を ADDF の仕組みとして支える:
   メインループの運転（判断・選択・検収・オーナー対話）と、実装・レビューのサブエージェント実行を
   別モデルに分離できるようにする
2. モデル指定を **フル ID でピン留めする手段**を明文化する（tier エイリアス `opus`/`sonnet`/`haiku`
   は将来のモデル世代に自動追従するため、特定世代に既知の問題がある場合はダウンストリームが
   意図的に固定できる必要がある）
3. モデルの得意不得意に応じた **エスカレーション基準**（実装が難航したときの逃し方）を明文化する

## 現状の挙動

- `.claude/agents/*.md` の `model:` フィールドは全て tier エイリアス（`sonnet`/`haiku`）で、
  実装専任のサブエージェント定義が存在しない（実装は `/addf-dev` からメインループが直接
  Agent tool で都度アドホックに委譲している）
- モデルの得意不得意（推論バジェット・美的感覚・計画遵守能力等）に応じた役割分担の考え方が
  どこにも文書化されていない。運転手（メインループ）と実装者を意図的に別モデルにする発想が
  仕組み側に存在しない
- `opus`/`sonnet` 等の tier エイリアスは「その時点の最新版」に解決される。ある世代に
  ダウンストリームが把握している既知の不具合（ツール呼び出しの不安定さ等）があっても、
  エイリアス指定のままではその世代を意図的に避けられない

## 変更内容（項目・フェーズ）

### 項目1: `addf-implementer` エージェントの新設

- **対象**: `.claude/agents/addf-implementer.md`（新規、配布物）

実装専任のサブエージェント定義を追加する。既存の `addf-code-review-agent` 等と並ぶ配布エージェントとし、
以下を満たす:

- 起動直後に `/addf-knowhow-index` 相当を実行し、関連知見を把握してから着手する
- 委譲プロンプトで渡されたスコープ（対象ファイル・Plan・完了条件・検証コマンド）に厳密に従う
- `isolation: "worktree"` 経由で起動された場合、worktree 外のファイルを変更しない
- Stage 1（ビルド・Lint・テスト）を自走で完遂する。失敗時は原因分析→修正→再実行まで自分で行う
- コミットは呼び出し元の明示指示がある場合のみ行う（デフォルトは行わない — 委譲プロンプトが
  コミット可否を明示していない場合は変更のみで留め、報告に「コミット未実施」と明記する）
- 対応できなかった項目は理由付きで報告する（silent に諦めない）
- 実装フェーズ完了時の知見を `/addf-knowhow` で記録する（既存エージェント定義の「知見の蓄積」節に倣う）
- `model:` フィールドは tier エイリアスのデフォルト値 + コメントで運用方針（ピン留めの使い分け）を示す（項目2参照）

### 項目2: モデル指定のピン留め手段の明文化

- **対象**: `.claude/agents/addf-implementer.md` の model フィールドのコメント、
  項目3のガイド新設ファイル

tier エイリアス（`opus`/`sonnet`/`haiku`）とフル ID 指定の使い分けを明記する:

- エイリアスは「その時点の最新版」に自動追従する。通常はこれで十分
- ダウンストリームが特定世代に既知の問題（ツール呼び出しの不安定さ等）を把握している場合、
  `.claude/agents/*.md` の `model:` にフル ID（例 `claude-opus-4-7-...` の形。実際の ID 文字列は
  利用時点の公式ドキュメントを参照）を直接書いてピン留めできる
- ピン留めは一時的な回避策として位置づける（新版で問題が解消されたら見直す）。理由を
  コメントで残す運用を推奨する

### 項目3: モデル配分ガイドの新設

- **対象**: `.claude/addf/guides/model-allocation.md`（新規）

役割 × モデル特性のマッピング方法を扱う独立ガイドを新設する（speculative-development.md と
同様の「概観のみ、詳細は各所が正」の構成）。含める内容:

1. **配分の考え方**: 各ロール（メインループ運転手 / 実装サブエージェント / レビュー / UI・文言・
   ドキュメント / Plan 起草・難所）に求められる能力軸（推論の深さ・意図の汲み取り・計画遵守・
   美的感覚・速度）を挙げ、ダウンストリームが自分の評価に基づいて表を作れるようにする
   （固定式のスコアリングにはしない — 投機適性 3 区分・7 割共有原則と同じ「特徴ベース判定」
   思想を踏襲する）
2. **実現手段（3つ）**:
   - `.claude/agents/*.md` の `model:` frontmatter（エージェント単位で固定）
   - メインセッションの `/model` コマンド（ループ全体の既定モデルを切替）
   - `Agent` tool 呼び出し時の `model` パラメータ（1回限りのオーバーライド）
3. **エスカレーション基準**: 実装が難航した場合（同一サブタスクで実装 2 回失敗、または着手前の
   見積もりでステップ数が大きいと分かった場合）に、無理に押し切らず次のいずれかへ逃す:
   - 調査だけを委譲して Plan を厚くしてから再実装する
   - `.claude/addf/Questions.md` に置いてオーナー判断を仰ぐ
   - TODO に「高性能モデル待ち」等のラベルを付けて一旦退避する
4. **7 割共有原則の補正**: 推論バジェットが小さいモデルでメインループを運転する場合、
   自前で長い実装をせず必ずサブエージェントへ委譲する。閾値判定も保守側（`nervous` 寄り）に
   倒す運用を推奨する（CLAUDE.md 「迷ったときの作法」との接続注記）
5. 参考として、ダウンストリームの実運用評価表の実例（出典を明記した引用）を1つ掲載し、
   「このような表を自分のプロジェクトで作る」イメージを示す

`development-process.md` の基本原則または関連節から本ガイドへのリンクを1行追加する
（節そのものの追加は行わない — development-process.md は概説に留める既存方針を尊重する）。

### 項目4: ダウンストリーム向けテンプレート節の追加

「品質ゲート拡張（オプション）」と同様の構成で、「モデル配分ポリシー（オプション）」節を追加する。
