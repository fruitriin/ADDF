# Plan 0058: Dashboard の HTML 化とブラウザレビュー UI — crit.md 連携の検討（検討スタブ）

## 実装状況: 未着手

> 出典: オーナー発案（2026-07-16 対話セッション）。Dashboard を md から HTML にしたい。
> VitePress 的な markdown to html がベースで、折りたたみ・画像・ページ切り替えで
> 「レビューの詰まり解消」「進行中のタスク」を見やすくし、差分ビューアも欲しい。
> さらに https://crit.md/ の仕組み（またはその模倣）に乗ってコメントできるととてもよい。

## 分かっていること

- **現状の Dashboard 運用**: `.claude/addf/Dashboard.md` は unattended 自走の差分まとめ。
  ブートシーケンス 1.6 で「セッション冒頭に提示 → オーナー応答確認 → 削除」の一時ファイル。
  プレーン md のため、差分の多いセッション後は読み込みが重く、オーナーの確認行動
  （どれを承認しどれに指示を返すか）が構造化されていない
- **オーナーの要件（4点）**:
  1. VitePress 的な markdown → HTML がベース
  2. 折りたたみ・画像・ページ切り替え（「レビュー詰まり解消」ビューと「進行中タスク」ビュー）
  3. 差分ビューア
  4. crit.md の仕組み／その模倣に乗った行コメント
- **crit.md の調査結果**（2026-07-16 実調査。https://crit.md/ と GitHub tomasz-tomczyk/crit）:
  - AI コーディングエージェント向けの**ローカルファースト・レビューツール**。MIT ライセンス。
    `brew install crit` / `go install` で導入。サーバは 127.0.0.1 バインドで外部通信なし
  - `crit plan.md` で markdown をレンダリングし（markdown-it・構文ハイライト）、
    ブラウザ上で**行単位・範囲選択のインラインコメント**が付けられる
  - コメントは `~/.crit/reviews/` に JSON の構造化レビューファイルとして保存され、
    **エージェントがファイル読取で拾う**プロトコル（API なし・ロックインなし）。
    `crit comment <file>:<line> '...'` で CLI からも追加できる
  - **split/unified の diff ビュー**・ラウンド間 diff（round-to-round）・未解決スレッド管理を内蔵
  - Claude Code 公式プラグインあり（`claude plugin install crit@crit`、`/crit` で
    レビューループ自動化）。ブランチごとに独立したレビューファイル
  - → 要件 1・3・4 を単体でほぼ満たす。「模倣した仕組みを自作」する前に
    「**crit をそのまま使う**」が有力な選択肢
- **設計方向の候補**:
  | 案 | 内容 | 見立て |
  |---|---|---|
  | A | crit を素直に採用（`crit Dashboard.md` ＋レビューファイルをブートシーケンスで読む） | 「作らない」がベース。ADDF 側の仕事は配線のみ。ページ切り替え等のビュー要件は crit の UI に従う |
  | B | VitePress で自作（ローカル dev サーバでダッシュボードサイトを生成） | ビューの自由度最大（ページ切替・折りたたみ・系統樹同居）。ただしコメント基盤と diff ビューアを自作する羽目になり重い |
  | C | ハイブリッド: 表示・ナビゲーションは VitePress、コメントと diff は crit | 両取りだが2系統の運用。段階導入（A で始めて不足分だけ B を足す）に分解できる |
- **ADDF 既存資産との接続**:
  - Plan 0039 の VitePress サイト骨格（`docs/.vitepress`）が既にある。ただしあちらは
    **公開ドキュメント**（GitHub Pages 向け）で、Dashboard は**ローカル・一時・非コミット**の
    性質のため、同一サイトに同居させるかは要判断
  - crit は外部 Go バイナリ — ccchain（Plan 0040）で確立した「対話セッションで
    ドッグフーディング → Behavior.toml オプトイン + optional/ テンプレートで配布」の
    パターンがそのまま流用できる。導入時の許可の取り方（外部コード取得・実行は
    AskUserQuestion で一段階ずつ明示確認）も Plan 0040 フェーズ1 の実績に従う
  - **crit のレビューファイルは Questions.md の非同期ループと同型**（エージェントが置く →
    オーナーが気が向いたとき応答 → 次セッションが反映）。ブートシーケンス 1.6（Dashboard
    提示）に「crit レビューファイルの未解決コメント確認」を組み込めば、Questions.md の
    Answer 記入が「ブラウザで行コメント」に置き換わる GUI 版になる
  - Plan 0056（Plan 系統樹）の Mermaid 派生ビューは、本 Plan のダッシュボードの1ページに
    載せる先として自然（クリティカルパス＝オーナー判断待ちの可視化はオーナー好感触あり）

  （`/addf-dashboard` 等）か。Dashboard.md の「確認後削除」ライフサイクルと
  crit レビューファイル（`~/.crit/reviews/`・リポジトリ外）の整合をどう取るか
- 配布形態: 本体ドッグフーディング先行（フェーズ分割）か、最初から
  optional/ テンプレート化まで見据えるか
