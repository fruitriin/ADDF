## Context

Fable 5 プロンプティングガイドの評価（2026-06-11）で、現行 `turn-reminder.sh` の
「コンテキストが大きくなっています」という注入文が、ガイドの警告する
**「コンテキストバジェットへの言及がモデルの早期切り上げ（作業縮小・新セッション提案）を誘発する」**
パターンに該当しうると判明した。

ただし対処にあたって2つの前提がある:

1. **turn-reminder は Fable 専用ではない。** ADDF はダウンストリーム配布物であり、
   Opus / Sonnet / Haiku を含む複数モデルで使われる。Fable の癖だけに最適化してはいけない
2. **能動コンパクション自体は引き続き必要。** 残量が実際に少なくなったら、
   システムコンパクション（制御不能な要約で文脈が失われる）に巻き込まれる前に、
   エージェント自身が知見記録・日記更新を能動的に済ませるべきである

つまり問題は「リマインダーをやめるか続けるか」ではなく、現行実装が
**2つの異なる関心事を「ターン数」という単一の粗い代理指標で混ぜていること**にある。

## 関心事の分離

### 関心事A: 知見の取りこぼし防止（ターンベースで継続）

セッションが長くなると knowhow 記録を忘れる。これはコンテキスト残量と無関係に
「長く働いたら棚卸しする」という習慣の問題であり、ターン数ベースで十分。モデル非依存。

**対応**:
- 文言から「コンテキストが大きくなっています」等の**残量への言及を削除**する
- Fable 5 ガイドの推奨に従い**安心文を追加**する:
  「これは作業を切り上げる指示ではない。コンテキストは十分残っている。
  知見を記録したらそのまま作業を継続すること」
- 趣旨を「コンパクション対策」から「知見の定期棚卸し」に純化する

### 関心事B: 能動コンパクション（実コンテキスト使用量ベースで新設）

ターン数はコンテキスト使用量の粗い代理指標でしかない（1ターンの重さは作業内容で桁が変わる）。
システムコンパクション前の能動整理は、**実際のトークン使用量**に基づいて発火すべき。

さらに、発火閾値は「コンテキストウィンドウの公称サイズ」ではなく
**「そのモデルが実質的に扱える器（実効コンテキスト）」**に基づくべきである:

- Opus は 1M 設定でも 250k 前後から徐々に文脈の取り違えが増えるため、
  200k 前後でのリカバリー（知見記録→整理）が実用上有効だった（オーナー観測）
- Fable 5 は実効的に扱える器が大きく拡大している可能性がある。
  確認でき次第、閾値を引き上げられる設計にする

**対応**:
- UserPromptSubmit フックの stdin JSON から `transcript_path` を取得し、
  transcript JSONL の直近 assistant メッセージの `usage`
  （`input_tokens` + `cache_read_input_tokens` 等）から現在のコンテキスト使用量を推定する
- 閾値超過時のみ「システムコンパクションに巻き込まれる前に、knowhow 記録・日記更新を
  済ませること。済んだら作業を継続してよい」というリマインダーを注入する
- この注入文にも安心文を含める（残量逼迫の事実は伝えるが、
  「作業を縮小せよ」ではなく「整理を済ませてから続行せよ」という行動指示にする）

### 閾値の設定化（addf-Behavior.toml）と判断の分担

フックはウィンドウ variant（200k / 1M）を判別できないが、モデル自身は知っている
（調査結果 2.5）。そこで**判断を分担する**:

- **フックの仕事**: 観測事実の注入。実測トークン数と、モデル名に対応する
  「実効コンテキストの目安」（toml の表）を添える
- **モデルの仕事**: 自分の variant・作業状況と突き合わせて、
  能動コンパクション（知見記録 → 整理）を実行するか判断する

これは関心事Aの修正と同じ原理（固定文言の決め打ち=詐欺をやめ、
根拠のある事実を渡して判断させる）の徹底でもある。

```toml
[context-reminder]
# リマインダー発火閾値（実測トークン数）。これを超えたら事実注入を開始する
threshold_tokens = 180000
# モデルごとの「実効コンテキストの目安」（モデル名の部分一致）。
# 公称ウィンドウではなく、文脈の取り違えが増え始める観測値に基づく。
# 注入文に目安として添えられ、判断はモデル自身が行う
[context-reminder.effective-context]
"opus" = 200000    # Opus 1M: 250k 前後から劣化が観測されたため手前の 200k
# "fable" = 未計測。計測でき次第追加（1M の実効器が広い可能性あり）
```

- 閾値設計の安全性: 200k 版モデルは公称上限の手前でシステムコンパクションが
  先に走るため、1M 前提の高い目安値を持っていても誤動作しない（届く前に圧縮される）。
  目安値は実質的に 1M 版でのみ意味を持つ
- 注入文の例: 「現在のコンテキスト使用量は実測で約 N トークン。
  あなたのモデル（<model名>）の実効コンテキストの目安は M トークン。
  目安を超えている／近い場合は、知見記録・日記更新を済ませてから作業を継続すること。
  これは作業を縮小・切り上げる指示ではない」
- Fable 系の実効器計測は本計画のスコープ外。値が定まったら toml に1行足すだけ

## 調査結果（2026-06-11 確認済み）

1. **`transcript_path` はフック入力に含まれる**（確定・公式ドキュメント）。
   UserPromptSubmit の stdin JSON: `session_id` / `transcript_path` / `cwd` /
   `permission_mode` / `hook_event_name` / `prompt`。
   モデル名・コンテキスト使用量はフック入力には**含まれない**
   （出典: https://code.claude.com/docs/en/hooks.md）
2. **transcript JSONL から usage を読める**（確定・実機検証）。
   `type: "assistant"` エントリの `message.usage` に
   `input_tokens` / `cache_read_input_tokens` / `cache_creation_input_tokens` があり、
   3つの合算が現在のコンテキスト使用量の良い近似になる
   （実測例: 2 + 93627 + 3834 ≈ 97k）。
   **モデル名も `message.model` から取れる**（実測: `claude-fable-5`）ため、
   モデル別オーバーライドは transcript 経由で実装可能
2.5. **ただしウィンドウ variant（200k / 1M）は transcript から判別できない**
   （確定・実機検証）。`claude-fable-5[1m]` セッションでも `message.model` は
   素の `claude-fable-5` と記録される。フック単独で判別する補助手段は
   「実測 usage が 200k を超えていれば 1M 確定」のみ。
   一方、**セッション内のモデル自身は variant を知っている**
   （システムプロンプトに `[1m]` 付きモデルIDが渡される）→ 設計に反映（下記）
3. **コンテキストウィンドウの公称サイズ（分母）はフックからは取れない**
   （statusline スクリプトの `context_window.context_window_size` にしかない）。
   → 閾値は「残量%」ではなく**絶対トークン数**で持つ本計画の設計が正しい。
   実効器ベースの絶対値はそもそも公称サイズと独立なので、分母は不要
4. **コンパクション後は自己補正される**。直近 assistant の usage は
   圧縮後のコンテキストを反映するため、推定値は次の応答で自然に下がる。
   特別なリセット処理は不要

## 実装上の注意（調査で判明）

- transcript にはサブエージェントの応答（sidechain）が混ざりうる。
  メインチェーンの最後の assistant エントリを使うこと（`isSidechain` 等で除外）
- 第1応答前・圧縮直後など usage が未取得のタイミングがある。
  パース失敗・エントリ不在時は関心事Bを**静かに SKIP** する
  （誤発火より無発火が安全。関心事Aのターンベースの網は常に生きている）
- JSONL パースは python3 を使う（既存 addfTools が Python 前提のため依存は増えない。
  jq は新規依存になるので使わない）。大きい transcript は末尾だけ読む（tail 等）
- statusline は人間向け表示専用で、モデルへの注入はできない。
  リマインダー注入はフック経由が唯一の経路（本計画のアプローチで正しい）
