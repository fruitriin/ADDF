## 目的

Feedback.md を「書ける側（knowhow）に流れて書かれない」状態から、構造的に書きやすく・
回収されやすいチャンネルにする。

## 現状の挙動

- Issue 起票時（2026-05）はフリーフォーマットで、書式の手がかりがなかった
- その後の運用で「オーナーフィードバック / 問題の記録 / 改善アクション / 完了済み」の
  セクション構造が自然発生しているが、**テンプレート・運用ルールとしての明文化はない**
  （downstream の初期状態ではセクション見出しだけの薄いファイルが配布される）
- Issue 内の分析が挙げる構造問題:
  1. knowhow と Feedback の振り分け基準が曖昧（knowhow に書いた瞬間 Feedback に書かなくて
     いい気がする — 書ける側に流れる）
  2. Feedback.md の出口が遠い（knowhow は次 Plan の filter ですぐ使われるのに対し、
     Feedback は効果が見えにくい）
  3. 「問題の記録」は書くこと＝引き受けることに見え、心理コストが高い

## 変更内容（項目・フェーズ）

### 項目1: 記入フォーマットの明文化

- **対象**: `.claude/addf/Feedback.md` の冒頭「記録方法」節（本体・downstream 配布版の両方）
- Issue 提案の「起きた問題：/改善方法：」ペア書式を導入する。改善方法は空欄でも複数案でもよい
  （「書くこと＝問題の存在確定」で止まらず「サイクルを回すこと」として消化できる設計 —
  問題と解決をペアで書き、対応済みマークで達成が回る）
- 強制はしない（Plan 0016〜0020・0048 の趣旨に合わせ、書きやすくする方向のみ）

### 項目2: knowhow との振り分け基準の明示

- **対象**: 同上＋`/addf-knowhow` スキル本文の該当箇所
- 「再利用可能な技術知識 → knowhow / フレームワーク・プロセス自体への改善要望・未解決の問題 →
  Feedback.md」の一行基準を両側に記載し、「書ける側に流れる」を構造的に防ぐ

### 項目3: 出口の可視化（検討）

- Progress 運用ルール（ステップ11〜13）は既に「書かないと進めない」構造を持つ — 現状の
  運用ルールとの重複を確認し、足りない部分（対応済みマークの付け方・完了済み項目の削除
  タイミング）だけ補う
