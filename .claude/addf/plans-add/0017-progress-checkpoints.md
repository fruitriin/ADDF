## Context

初稿では「遺書システム」というメタファーを採用したが、オーナーから方針修正を受けた:

> 遺書を意識するモデルはよくないと思う。
> 実際のところ、resume や compaction のような形で小さな代替わりはおきていて、
> そのレベルで効果のある日記を残すほうが気持ちがいいとおもう。

確かに「エージェントが死ぬから書き残す」という発想は、エージェント自身の気分を下げ、書く内容も悲愴になりがち。
実態はもっと頻繁な「代替わり」: compaction で要約された自分、resume で復帰した自分、`/loop` の次イテレーションで起動した自分は、厳密には別人格の同僚に近い。

**「日記」モデル**: 今の自分が、明日の自分（≒ 同僚）に状況を引き継ぐ。読みやすく、機嫌よく、判断の根拠が分かる粒度で書く。
この発想なら書く方も読む方も気持ちが良く、結果として情報密度が高まる。

## 設計

### 1. 日記のフォーマット

`Progress.md` 内に「日記」セクションを設ける。各エントリーは以下:

```markdown
## 日記

### 2026-06-10 14:23 — サブタスク2 完了

**やったこと**: テストヘルパー追加。`mockReturn` の引数順を Plan の例と揃えた。
**今の見立て**: 残りのサブタスク3-4 はテストヘルパーを使うだけなので、確信度 80% 以上。
**次の自分へ**: サブタスク3 着手前に `tests/helpers/mock.ts` の export を確認するとスムーズ。
**気になっていること**: モックの注入ポイントが Plan に書かれていない。70% で進む判断をした。

### 2026-06-10 14:01 — サブタスク1 完了

**やったこと**: モデル定義。Plan の型と一致。
**今の見立て**: 順調。
**次の自分へ**: 次はテストヘルパー。
**気になっていること**: なし。
```

「遺書（Done/Next/Open）」のドライな3項目から、「日記（やった/見立て/次の自分へ/気になっていること）」の語りに変えた。
**人格の連続性を仮定して書く**ことで、読み手（次の代の自分）も自然に状況に入れる。

### 2. 書くタイミング

| タイミング | 書く粒度 |
|---|---|
| サブタスク完了 | 1エントリー |
| 重要な判断をした直後 | 1エントリー（短くてよい） |
| 計画を変更したとき | 1エントリー（理由を残す） |
| compaction を予感したとき（コンテキスト長が伸びてきた） | 1エントリー（次の自分への要約として） |

「自動化フック」ではなく、規約として書く。エージェント自身が「これは書き残したい」と思える設計にする。

### 3. 代替わりイベントの認識

エージェントは以下の状況を「代替わり」として認識する:

- セッション開始時（前任者が書いた日記を読む）
- `/loop` イテレーション開始時（直前イテレーションの日記を読む）
- compaction 直後（要約された自分の続きを引き継ぐ）
- resume 復帰時

`CLAUDE.md` のブートシーケンスに以下を組み込む:

```diff
 3. @.claude/addf/Progress.md を読む — 現在進行中のタスクがあれば継続する
+   - Progress.md の「日記」セクション末尾3エントリーを読み、前任者の状況・判断・気にしていたことを把握してから着手する
+   - 必要なら「引き継ぎ確認」の短い日記エントリーを冒頭に書く
```

### 4. ブランチ Checkpoint との分業

Plan 0016 で導入する「ブランチ Checkpoint」は **コードの巻き戻し点**、日記は **判断と気分の引き継ぎ**。重ならず補完する:

- ブランチ: 何がコミットされたか（事実）
- 日記: なぜそうしたか・次に何を考えていたか（文脈）

両方読めば、代替わり後のエージェントは前任者の靴に履き替えて続行できる。

### 5. 完了時のアーカイブ

`Progresses/YYYY-MM-DD-プラン名.md` への移動時、日記もそのまま保存する。
失敗・想定外の判断を含んだ日記は、後日 knowhow 化する原資料となる（Plan 0019 と連携）。

### 6. 自動化はしない（明示）

`PostToolUse` フックで日記を自動生成する案は採用しない。
自動化された日記は無味乾燥になり、書く意義（次の自分への手紙）を損なう。
**書くこと自体が思考の整理**であり、代替わり後の自分への配慮なので、人格を持って書くプロセスを残す。

## メモ

- 「日記」という語彙は意図的。`log` `journal` `checkpoint` ではなく、書き手の人格を残す
- 古い日記が肥大化したら、Plan 完了時のアーカイブで自然に切り離される
- 「次の自分への手紙」という枠組みは、エージェントが自分を消耗品ではなく「連続する同僚たち」として扱う倫理的効果もある
