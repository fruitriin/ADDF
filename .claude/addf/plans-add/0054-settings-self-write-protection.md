## 分かっていること

- 検出元の懸念: エージェントが `.claude/settings.json`（permissions 設定）や `.claude/hooks/`
  配下のフックスクリプトを自分自身で書き換えられると、deny/ask ルールやガードフックが
  無効化されうる（自己防衛機構の自己書き換え問題）
- Plan 0043 は「不便のない範囲で」の指針と両立させるのが難しいと判断し対象外にした:
  Claude Code の `Write`/`Edit` を settings.json・hooks に対して全面 deny すると、
  `/addf-init` によるセットアップや運用ルール更新（フィードバック反映等）が事実上できなくなる
- Plan 0043 が示唆したより精密な設計の方向性:
  - hooks ディレクトリのみ deny する
  - settings.json のみ deny する
  - write が発生した瞬間に diff を提示する（ask 相当だが自動承認しない）

## 設計案の整理（開発上の便利/不便 × セキュア/アンセキュア）

### 前提: どの案も「誤操作・無人自走の暴走防止」であって「悪意への防御」ではない

`Write`/`Edit` への deny をどう組んでも、`Bash` 経由の書き換え（`sed -i`・リダイレクト・
`cp` 上書き）は塞げない。Bash 側のガードは deny パターン（Plan 0043 項目1）や ccchain
（PreToolUse(Bash)、Plan 0040）という別レイヤーの仕事。本 Plan が想定すべき脅威は
「エージェントが善意のまま、文脈の薄い状態（compaction 後・unattended 自走・並行
セッション）で自己防衛機構を書き換えてしまう**事故**」に置く。したがって完了条件も
「完全な自己書き換え防止」ではなく「Write/Edit 経路での事故的書き換えに気づける・
止められる」に置くのが妥当。

### 評価する案

| 案 | 内容 |
|---|---|
| A | settings.json 系 + hooks/ への Write/Edit を全面 deny |
| B | hooks/ のみ deny |
| C | settings.json 系のみ deny |
| D | deny せず、書き換え時に diff 提示 + ask（`permissions.ask` で実装可・自動承認しない） |
| E | 部分保護（`permissions.deny` キーの削除・緩和のみブロックするフック検査） |
| F | 現状維持（保護なし・実害の実測をトリガーに再検討 = B2 判断と同型） |

### ADDF 本体開発での評価

本体では settings.json・hooks/ が**開発対象そのもの**（フック新設・配布テンプレートの更新・
sync-ccchain.py・/addf-permission-audit の提案反映）であり、書き換え頻度が高い。

| 案 | 便利/不便 | セキュア/アンセキュア | 備考 |
|---|---|---|---|
| A | ✗✗ 開発がほぼ止まる | ◎ | フック開発のたびに人間が deny 解除する羽目になる |
| B | ✗ フック開発は本体の主業務の一つ | ○ | destructive-git-guard 等の開発を阻害 |
| C | ✗ audit / sync 系が機能不全 | △（hooks 側が無防備） | |
| D | △ 変更頻度が高く ask が煩雑 | ○ | unattended では ask 不応答 = 実質 deny（安全側に倒れる） |
| E | ○ deny 緩和以外は自由 | △ 検査フック自体が自己書き換え対象という再帰問題 | 実装コスト大 |
| F | ◎ | ✗（ただし実害は未観測） | cron 並行実行の実績があるリポジトリではリスク漸増 |

2軸での位置づけ: **本体では「セキュアかつ便利」の象限に入る案が存在しない**。D が最も
近いが、変更頻度の高さで便利軸が削れる。

### ダウンストリーム開発での評価

ダウンストリームでは settings.json・hooks/ は**基本的に配布物**であり、日常的に触るのは
「独自権限の追加」程度。書き換えの正当な主体はほぼ /addf-init・/addf-migrate に限られる。

| 案 | 便利/不便 | セキュア/アンセキュア | 備考 |
|---|---|---|---|
| A | ✗ init / migrate が止まる | ◎ | migrate 時のみ解除する運用が別途必要 |
| B | ○ hooks はほぼ触らない | ○ ガードフック無効化を防げる | migrate のフック更新だけ手当てが要る |
| C | △ 独自権限追加（正当なカスタマイズ）を阻害 | △ | 本 Plan 起票時の懸念そのもの |
| D | ◎ 変更頻度が低く ask 負担が小さい | ○ unattended 自走中の変更は止まる | |
| E | ○ | △ | 実装・配布コストが便益に見合うか疑問 |
| F | ◎ | ✗ unattended 自走する DS ではリスク最大 | |

2軸での位置づけ: DS では **D（＋任意で B を重ねる）が「セキュアかつ便利」に最も近い**。

### 整理から得られる示唆

1. **本体と DS で最適解が異なる**（本体: F または D / DS: D ＋任意で B）。単一の設定を
   そのまま配布する発想を捨て、**保護強度を配布先で変える**のが自然
2. 配布方式は ccchain（Plan 0040）と同じ **Behavior.toml オプトイン + optional/
   テンプレート**のパターンが流用できる（DS が自分のリスクプロファイル〔unattended
   自走の有無〕に応じて選ぶ）
3. D の「unattended では ask が実質 deny になる」性質は、リスクが最大の状況（無人自走）で
   保護が最も強くなる**良い非対称**。対話セッションでは人間が diff を見て通せるので
   「不便のない範囲で」とも両立しやすい
4. D は `permissions.ask` に `Edit(.claude/settings.json)` 等を置くだけで実装でき
   フック不要。ただし ask ルール自体も settings.json 内にあるため「書き換えで保護を
   外せる」再帰は残る（事故防止目的なら許容範囲。悪意対策でないことは前提節のとおり）
