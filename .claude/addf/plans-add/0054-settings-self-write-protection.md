## 分かっていること

- 検出元の懸念: エージェントが `.claude/settings.json`（permissions 設定）や `.claude/hooks/`
  配下のフックスクリプトを自分自身で書き換えられると、deny/ask ルールやガードフックが
  無効化されうる（自己防衛機構の自己書き換え問題）
- Plan 0043 は「不便のない範囲で」の指針と両立させるのが難しいと判断し対象外にした:
  Claude Code の `Write`/`Edit` を settings.json・hooks に対して全面 deny すると、
  `/addf-init` によるセットアップや運用ルール更新（フィードバック反映等）が事実上できなくなる
- Plan 0043 が示唆したより精密な設計の方向性:
  - hooks ディレクトリのみ deny する
  - settings.json のみ deny する
  - write が発生した瞬間に diff を提示する（ask 相当だが自動承認しない）

## 未解決の問い

- 上記3方向性のどれを採用するか、あるいは別の設計（例: 特定キー〔`permissions.deny`〕のみ
  書き換え不可にする部分保護）が現実的か
- 「不便のない範囲で」との両立をどう測るか（Plan 0043 の Q3 解消時と同様、事後観測方式で
  段階調整する運びでよいか）
- ダウンストリームへの配布時、この保護機構自体がダウンストリームの正当なカスタマイズ
  （settings.json への独自権限追加等）を妨げないか
