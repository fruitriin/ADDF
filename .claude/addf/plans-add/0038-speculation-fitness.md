## 目的

投機（addf-speculate）に「向くタスク／向かないタスク」の判定基準を導入し、
向かないものを正規ルートへ導くフォールバックと、大改造の実施タイミング（投機在庫ゼロの窓）を
検出してオーナーに選択を迫る仕組みを作る。

## 項目1: 投機適性の判定基準（選定手順への組み込み）

現行の選定基準「直交概念であること」に、明示的な**除外基準（投機アンチパターン）**を加える:

| 区分 | 特徴 | 例 |
|---|---|---|
| 投機向き | 直交・局所的・失敗を捨てられる・本流と衝突しにくい | 注記追加、独立スキルの試作、lint 1本 |
| 投機不向き | 全域に触れる・参照の一貫性が必要・並走ブランチを全滅させる | ディレクトリ移動（0037 型）、一括リネーム、テンプレート同期ペアの変更 |
| 投機禁止 | 不可逆・オーナー判断が本質 | リリース、削除系、外部公開 |

- 判定は選定時に行い、判定理由を Worktrees.md（または Dashboard）に1行残す
- 「不向き/禁止」は数値スコアではなく特徴ベースの判定とする（直交性の見積もりと同様、
  最終判断はケースバイケースでよい — 7割共有原則の思想と整合）

## 項目2: 不適合タスクのフォールバック導線

投機候補として浮かんだ概念が「不向き/禁止」に該当したら、**捨てずに正規ルートへ流す**:

1. Plan 草案を起こして `TODO` に登録する（= /addf-dev の通常タスクとして実施される導線）
2. 大改造級（0037 型）なら Plan に **`execution_style: one-shot`（一発通し切り）マーカー**を
   付けて起案する。マーカーの意味は 0037 の実施様式（事前清算・リハーサル前置・
   単一セッション完走・巻き戻し優先）を指す — 定義の単一ソースは guides 側に置き、
   Plan からは参照する
3. Dashboard / 日記に「投機不適合のため Plan 化した」ことを記録する（silent に消さない）

## 項目3: 大改造の窓の検出と選択提示

投機在庫（開発中・採否判断待ち・持ち越し・Pending）が**ゼロになった瞬間**は、
大改造の事前清算条件（0037 実施様式）が自然に満たされている窓である。

- **検出**: addf-speculate の発動ガード（アイドル検出）に「reconcile check の在庫ゼロ ∧
  TODO に one-shot マーカー付き未着手 Plan が存在」の判定を追加する
- **選択を迫る**: 検出したら次の投機を始める前にオーナーへ提示する:
  - interactive: AskUserQuestion「今が窓です。<Plan 名> を実施しますか / 投機を続けますか」
  - relaxed / unattended: Questions.md に質問を置く。**このとき窓を保つため、応答待ちの間は
    新規投機を開始しない**（投機を再開すると窓が閉じ、次の窓がいつ来るか分からなくなる）
  - 応答待ちのまま N サイクル（既定: 3）経過したら、窓の保持を諦めて投機を再開してよい
    （自走を無期限に止めない。再開したことは Dashboard に記録する）
- one-shot Plan の実施自体は常にオーナーの明示応答が起点（0037 実施様式の
  「オーナー同席推奨」を継承。窓の検出は提案までで、自動着手しない）

## 設計メモ

- 判定基準・フォールバック・窓検出はすべて addf-speculate スキル本文と guides
  （投機運用ガイド）の記述で実現し、新規スクリプトは原則作らない
  （在庫ゼロ判定は既存の speculate-reconcile check の出力を使う）
- `execution_style: one-shot` マーカーは 0033 項目4 / 0035 項目3で整備する
  PlanTemplate・フロントマター運用に載せる（依存: PlanTemplate が先にあると綺麗だが、
  マーカー単体はテンプレート無しでも運用開始できる）
- 窓の検出は「毎回の発動ガードに1判定足す」だけで、常駐監視は作らない
