## 分かっていること

### 現状

- ignore ベースラインは upstream が決めた単一の `.gitignore` ADDF ブロックを
  全プロジェクトに配布している。`.claude/commands/*.exp.md` は ADDF 本体でも
  ignore されており、本体ローカルに7ファイル存在するが1つもコミットされていない
- CLAUDE.md「並列実装方針」の「worktree 起動後に `.claude` をコピーする」運用は、
  「ignore されたファイルが隔離環境に存在しない」という同じ症状への既知の回避策。
  Claude Code Web / cloud で動かない問題は、この回避策すら打てない環境に現れた
  同一構造の一般化であり、新しい問題ではない

### 整理の3軸（2026-07-16 対話より）

1. **ファイルの性質**（そのファイルは「誰の状態」か）:

   | 性質 | 例 | 素直な扱い |
   |---|---|---|
   | 配布物 | スキル定義・テンプレート・hooks | コミット（upstream が所有） |
   | プロジェクト共有知 | knowhow・plans・Questions・Feedback | コミット |
   | 実行時ステート | .turn-count・logs・Dashboard.md | ignore |
   | 経験・最適化された状態 | .exp.md・.ccchain.local.conf・settings.local.json | **曖昧ゾーン — 軸2で答えが変わる** |

2. **開発形態**: upstream（ADDF 本体）/ ダウンストリーム個人 / ダウンストリームチーム
3. **実行環境の可搬性**: ignore ファイルは Claude Code Web・cloud・worktree 隔離に存在しない

### 具体例の位置づけ

- **`.exp.md`**: upstream ではドッグフーディングの産物であり、コミットすると配布物
  （スキルの初期状態）を汚す → ignore が正しい。ダウンストリームでは「このプロジェクトで
  踏んだ分かれ道の目印」というプロジェクト共有知に近い → コミットが自然。
  **upstream と DS でベースラインが異なるべき最初の具体例**
- **チームの安定/個人分離**: ADDF は既に **local ペア慣習**を持つ
  （settings.json / settings.local.json、.ccchain.conf / .ccchain.local.conf）。
  「安定共有物＝コミット、個人最適化＝`.local` サフィックスで ignore」を exp にも
  適用しうる（`.exp.md` 共有 + `.exp.local.md` 個人。ただしファイル増殖コストと相談）
- **個人開発 × Web**: コミットのデメリット（レビューノイズ・衝突）がほぼゼロ →
  ignore を実行時ステート最小限にして可搬性を最大化するのが便利

### 設計の方向性（たたき台）

- `addf-Behavior.toml` にプロファイル宣言を置く（例: `profile = "upstream" | "team" | "personal"`）
- `addf-init` / `addf-migrate` がプロファイルに応じて `.gitignore` の ADDF ブロックを生成する:
  - **upstream**: 現状どおり（exp も ignore）
  - **team**: exp はコミット、`.local` 系のみ ignore
  - **personal**: ignore 最小（実行時ステートのみ）— Web/cloud 可搬性を最大化
- Plan 0040（Behavior.toml オプトイン + optional/ テンプレート配布）・Plan 0054
  （保護強度を配布先で変える）と同じ「配布時可変」設計パターンの一般化として一貫させる

## 一本化の理由（分割しない判断）

- `.exp.md` の「コミットすべきか」はプロファイル概念なしでは答えが出ない
  （upstream/DS個人/DSチームで答えが変わる、が整理の結論）。先行して exp だけの
  Plan を作ると、そこでアドホックな切り替え機構を発明し、後続のプロファイル Plan が
  それを作り直す二度手間になる
- 実装の絡む中心が単一（`.gitignore` ADDF ブロックの生成と、それを検査する lint）。
  分割しても同じファイル・同じ lint を両 Plan が触ることになり、並行実行競合の
  リスクだけ増える
- exp.md は Plan 内の**フェーズ1（最初の具体例・検証ケース）**として扱えば粒度は保てる

## 未解決の問い

- プロファイルの粒度: 3値（upstream/team/personal）で足りるか。team と personal の
  差分は「exp をコミットするか」以外に何があるか（ファイル別の個別オーバーライドを
  許すか、それは複雑化しすぎか）
- `.gitignore` ADDF ブロックは Plan 0052 で lint（非対称検知）の検査対象になった。
  ベースラインを可変にする場合、lint がプロファイルを読んで期待値を切り替える必要が
  ある — ここが実装上最も絡む箇所。lint 側の期待値表をどこに単一ソース化するか
- 既存ダウンストリーム（taskbar.fm・wardrobe-test 等）のマイグレーション:
  プロファイル未宣言の既存プロジェクトのデフォルトをどれにするか
  （現状維持 = upstream 相当が安全側だが、DS にとっては不便側でもある）
- exp.md をコミット対象にした場合、`addf-migrate` のスキル更新が DS ローカルの
  exp 蓄積を上書きしないか（スキル本体と exp の分離が守られているかの確認）
- チーム文脈の `.exp.local.md` ペアを初期スコープに含めるか、必要が観測されてから
  追加するか（B2 判断と同型: 実害・実需の実測をトリガーにする選択肢）
