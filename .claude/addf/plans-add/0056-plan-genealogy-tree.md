## 分かっていること

- **系統樹はデータとして既に半分実在する**。線形テーブル（TODO.addf.md）の裏で、Plan 間の
  関係は既に系譜的な語彙で記録されている:
  - 派生（切り出し）: Plan 0053 のレビュー → Plan 0054 起票（Progress 運用ルール7
    「主題外は別 Plan に切り出す」は系統樹の分岐を生むルールそのもの）
  - 回収（吸収）: Plan 0026 の残り → Plan 0043 に回収
  - 剪定: Plan 0045 は実測（EnumaElish・wasurenainder）に基づく「意図的な不採用として決定」。
    Plan 0044 も案A採用＝案Bを証拠付きで剪定
  - 復活: Plan 0054 の出典欄が実例 — Plan 0026 で検出 → 0043 で明示的にスコープ外（剪定）→
    放置 → 0053 の doc-review で発掘 → 0054 として復活
  - 投機: Plan 0049 は worktree で投機実装 → 後日 main へ回収・採番変更
- `PlanTemplate.md` には「関連 Plan」セクションが既にあり、knowhow と同じ双方向リンク原則を
  宣言済み。欠けているのは (a) エッジの型分類、(b) グラフとしての描画、の2つ
- **エッジ型の最小セット案**（実績のある関係のみ。型を増やしすぎると記入コストと lint
  ノイズが増える — B2 保留時の「悩みどころ・ノイズは増えてほしくない」というオーナー感性）:

  | エッジ型 | 意味 | 実例 |
  |---|---|---|
  | `derived-from` | レビュー・実装中の発見から切り出された | 0053 → 0054 |
  | `absorbed-into` | 残タスクが別 Plan に回収された | 0026 → 0043 |
  | `pruned` | 証拠付きで不採用（理由と証拠を必須にする） | 0045（実測） |
  | `revives` | 剪定済みの枝を再起票 | 0054 revives 0026-Critical |
  | `blocked-by` | 依存・オーナー判断待ち | 0040 フェーズ4 ← フェーズ3 |

- **剪定枝は「死んだノード」ではなく「復活トリガー付きの休眠ノード」**。ADDF には既に
  「実害の実測をトリガーにする」判断様式が繰り返し現れている（B2 保留・Plan 0043 の
  事後観測方式・Plan 0054 の着手トリガー）。剪定ノードに `pruned(理由, 証拠, 復活条件)` を
  持たせると、復活検討時に問うべきことが「前提が変わったか？」の一点に圧縮される。
  代替わり日記が「文脈」、チェックボックスが「事実」を残すのに対し、系統樹は
  「決定の地形」を残す第三の層。不採用計画が「失敗の記録」ではなく「探索した証拠」として
  一級市民になるのは Plan 0016〜0020 の趣旨（責めない）とも整合する
- **このリポジトリでのクリティカルパスの実態は「オーナーの決断キュー」**。Plan 間の技術的
  依存は薄く、真のブロッカーはオーナー判断待ち（0048=Q6・0030 branch protection・
  0039 フェーズ3・0054 方向性選択）。系統樹の葉のうち `blocked-by: owner` の枝を強調する
  だけで、Questions.md と TODO 状態注記に散らばる「待ち」が一望できる。Dashboard.md
  （unattended 差分まとめ）とも相性が良い
- **実装方針の骨子**（詰めは標準テンプレート昇格時）:
  - TODO テーブルは置き換えない。線形テーブルは lint（lint-plan-status.py）と状態管理の
    真実源として維持し、系統樹は Plan の「関連 Plan」セクション＋メタデータから**生成する
    派生ビュー**にする（CLAUDE.repo.example.md の単一ソース化の教訓に従う。手書きの
    系統樹図は即ドリフトする）
  - 描画は Mermaid `graph` で十分（VitePress サイト＝Plan 0039 に載せられる）
  - 整合性検査は knowhow 双方向リンク lint（セクション8）と同型のため設計資産を流用できる。
    既知の罠も同じものを踏む想定（`## 実装状況:` ヘッダの1行制約・エッジ追記が自分の新設
    lint に引っかかる Plan 0052 型の自己言及問題）
- **発展の面白い可能性**: 新 Plan 起票時に addf-knowhow-filter と対になる「系譜フィルタ」—
  起票しようとしている内容が過去の剪定枝と類似していないかを照合し、「その枝は剪定済み。
  復活条件はこれ」と返す（「もう考えたな」の自動化）。/addf-plan-audit が既に行っている
  「埋没発掘」の常設ビュー化とも言える
