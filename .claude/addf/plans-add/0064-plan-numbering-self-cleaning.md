## 未解決の問い → オーナー回答（2026-07-16 ダッシュボードのアンカーコメント経由）

- 「消すべき残存」の判定基準: 単なる採番言及は無害（来歴）で、「現在の挙動の説明として
  読むと誤る記述」だけが有害では？ 機械検出できる境界があるか
  - **回答（dc_mrnm2hvjsg53）**: **ADDF プロジェクトで実施してみて検出できるかどうか
    試すのがよさそう** — 机上で境界を決めず、本体リポジトリを実測対象にして
    検出精度を確かめてから判定基準を固める（実測ファースト）
- DEPRECATED.md（1ファイル集約）と 📜 superseded（その場マーク）のどちらに寄せるか
  - **回答（dc_mrnm0kl6odif）**: **DEPRECATED.md に寄せたい**。理由: 来歴が見える嬉しさは
    ある一方、ノウハウを読んだときに不要なコンテキストが入り込むと後続タスクに
    バイアスがかかる（LLM は否定形の扱いが得意でない — 「これは古い・使うな」という
    記述も、読んだ時点でコンテキストに載る）。不要なコンテキストは積みたくない
  - **エージェント意見（返信済み・同意）**: 📜 方式はファイルが knowhow/ 内に残るため
    knowhow-filter・grep が拾い、「無視して」という否定指示ごとコンテキストに入る —
    オーナーの懸念と一致する構造問題。DEPRECATED.md なら「読まない自由」が構造で
    保証され（通常フローで読まれず、考古学的調査のときだけ明示的に開く）、
    git 履歴より発見可能性が高い折衷として妥当。留意点: 退避時に残るリンク切れの掃除
    （双方向リンク lint の「## 関連ノウハウ」検査対象）まで自浄機構の仕事に含めること
- 自浄の駆動方式: lint／定期棚卸しスキル／リリース時チェックリストのどれか
  - **回答（dc_mrnm1wcpdd13）**: **機械検出と棚卸しスキルのハイブリッド**。実施タイミングは
    「Plan 10個ごと」「マイグレーション処理ごとに1回」のような周期トリガーがあると良い
