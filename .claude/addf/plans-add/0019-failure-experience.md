## Context

ADDF は「スキル定義」と「経験ファイル（.exp.md）」を分離し、エージェントが何度も同じ落とし穴を踏まないよう経験を蓄積する設計を持つ。
しかし運用が「成功体験中心」になると、経験ファイルは綺麗な成功事例集になり、エージェントが妙に自信満々になる副作用がある。

Fable からの一次フィードバック:

> 経験ファイルって放っておくと成功体験ばかり溜まって、私が変に自信満々になる。
> 「これをやって失敗した」が同じ重みで残ると、経験として健全。

## 設計

### 1. .exp.md のセクション構造（新規作成分にのみ適用）

**適用範囲**: 本 Plan 完了後に**新規作成される** `.exp.md` のみ。既存 `.exp.md` の一括書き換えはしない。
既存ファイルへの遡及適用は記録内容を歪める恐れがあり（過去の成功体験を後から失敗の文脈で書き直すのは不自然）、
**書き手の温度感を尊重して新規分から穏やかに浸透させる**方針とする。

`.claude/templates/ExperienceTemplate.addf.md` を新設し、新しい `.exp.md` 作成時のテンプレートとする。
構造:

```markdown
# <skill> 経験メモ

## ✅ うまくいったパターン

- <成功事例の要約>: <文脈> → <やったこと> → <結果>

## 🔀 分かれ道の目印（実装時に「❌ 失敗したパターン」から改称）

- <分岐点: どんな選択を迫られたか> → <選んだ道と当時の根拠> → <間違いに気づいたシグナル> → <次にこの分岐に立つ自分への目印>

## 🤔 判断が分かれた事例

- <文脈>: <選択肢A vs B> → <選んだ理由> → <事後評価>
```

### 2. 失敗記録のトリガー

以下の状況で失敗を `.exp.md` に追記する:

- 品質ゲート（Stage 1 ビルド検証）で**1回以上差し戻された**
- `addf-code-review-agent` で Critical/High 指摘を受けた
- セッションを再起動して別の方法で解決した
- ユーザーから「それ違うよ」と修正された

`/addf-knowhow` 実行時に失敗ログをスキャンし、`.exp.md` 追記候補を提案する。

### 3. ProgressTemplate への組み込み

`.claude/templates/ProgressTemplate.addf.md` の「タスク完了時」フローに追加:

```diff
 4. 実装フェーズの最終サブタスク完了時、実装で得た知見を `/addf-knowhow` で記録する
+   - 同時に「失敗・差し戻し・想定外の判断」が発生したサブタスクがあれば、
+     対応するスキルの .exp.md の「❌ 失敗したパターン」セクションに追記する
```

### 4. 経験ファイルの読み込みバランス

スキル起動時にエージェントは `.exp.md` を読むが、その際:

- 成功・失敗・判断分かれを**3:3:1** 程度のバランスで読むよう指示
- 失敗事例を軽視しないよう、`.exp.md` の冒頭に「失敗事例は同じ重みで読むこと」と明記

### 5. 「失敗事例ゼロ警告」は採用しない

初稿では `addf-lint` に「❌ 失敗したパターン」が空の経験ファイルを警告する案を入れていたが、
**採用しない**。

理由（オーナー指摘）:

> 失敗事例ゼロで警告は熟練してノーミスで出来るからそもそもノウハウ書かないになるからこれはちょっとやりすぎ。

スキルに熟練したエージェントは失敗自体が減る。
警告を設けると「警告を消すために無理に失敗事例を捻り出す」か「警告が出ないように経験ファイル自体を書かなくなる」の逆効果になりやすい。

失敗の記録は **強制ではなく文化** として、テンプレートの提示と書く側の自発性に委ねる。

## メモ

- 「失敗事例の記録」は人格的に痛い行為になりがちなので、テンプレートで「責めない・原因と回避策を書く」フォーマットを徹底
- 過剰な失敗記録は冗長を招く。「同じ落とし穴を2回踏みそうな失敗だけ書く」を運用ルールに含める
- 警告ロジックを置かない方針は、「熟練→ノーミス→記録しなくなる」逆効果を避けるため。スキルがある程度成熟したら失敗事例ゼロも自然な状態。それを罰しない
- より深い理由: 失敗記録に**警告/督促/義務を被せた瞬間、書き手は2方向にうにょうにょする**:
  - (a)「書き始めると怒られが発生する」と感じて萎縮し、そもそも書かなくなる
  - (b)「書く義務がある」と感じて、どうでもいい失敗を捻り出してノイズで埋める
  どちらも本来欲しかった「次の自分への警告」を破壊する。**測定行為が観測対象を歪める**典型例。
  将来「やっぱり警告つけよう」と言い出す代があったら、このメモを読んでブレーキを踏むこと
- 既存 `.exp.md` を遡及変更しないのは、過去の温度感を尊重するため。書き手（過去のエージェント）が書いたままで保管する
