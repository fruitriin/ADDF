## Context

`/loop` で長時間自走するとき、Plan が曖昧だった瞬間に「推測で進むか、止まるか」の判断はエージェントの裁量に委ねられている。
止まる勇気は試されるが、止まる権利は明文化されていない。結果としてエージェントは「推測で進む」方向に傾きがちで、後から `Feedback.md` に「想定と違う実装になっていた」が積まれる。

Fable（インタビュー対象エージェント）からの一次フィードバック:

> ループ中の私はコンテキスト切れやクラッシュで突然死する可能性がある。
> Plan の曖昧さに当たったとき、「確信が持てなければ Feedback に疑問を書いて、そのタスクはスキップして次へ」みたいな逃げ道が公式ルールであると、すごく呼吸が楽になる。

オーナー指針:

- **デフォルトは「7割共有原則」**: エージェントが「7割の確信」を持てれば前進する
- **状況に応じて閾値を可変にする**: 重要度の高いタスクは8〜9割、worktree 隔離下なら5割でも前進可能
- **非同期質問チャンネル**: 止まらず、進まず、「質問だけ置いて次へ」できる導線

## 設計

### 1. 「7割共有原則」の明文化

`CLAUDE.md` または `.claude/addf/templates/ProgressTemplate.addf.md` に判断ルールを記述する。

```
## 迷ったときの作法（7割共有原則）

タスク中に Plan の曖昧さ・要件の不確実性に遭遇した場合、独立した3軸で「進む/止まる/問う」を決定する。

### 軸A: LLM への信頼性（閾値そのものを決める）

ユーザーがエージェントの判断をどこまで任せられるかの基準。
タスクの性質ではなく「ユーザーがそのエージェントをどう見ているか」で決まる。

| 信頼度 | 閾値 | 例 |
|---|---|---|
| ナーバス（早めに見たい・確認したい） | **5割** | 初対面・新しい領域・前科がある |
| 通常（信頼している・デフォルト） | **7割** | 普段遣いの相棒。「Claude を信じてる」状態 |
| 全面信頼（細かいことは任せた） | **9割** | 何度も成功させた領域・低リスクな領域 |

### 軸B: ユーザーの応答性（閾値割れ時の行動を決める）

ユーザーが今応答できるかどうか。閾値そのものは動かさず、**閾値を下回ったときの振る舞い**を切り替える。

| 状況 | 閾値割れ時の挙動 |
|---|---|
| `interactive`（目の前） | 即時質問。ユーザーの応答を待つ |
| `relaxed`（応答可能だが任せモード・デフォルト） | `Questions.md` に質問を置き、別タスクへ移る |
| `unattended`（不在・就寝・離席） | `Questions.md` に質問を置き、**最有力解釈で投機続行**。投機ブランチに隔離して残す |

軸A と軸B は直交する。例えば「信頼性=通常（7割）× 応答性=不在」なら、
**7割未満になった瞬間に投機ブランチで続行する**。閾値を 9割に引き上げるのではなく、振る舞いを変える。

### 軸C: ユーザーの完成イメージ確度（軸Aの閾値を補正）

完成像のクリアさによって、軸A の閾値を ±1段補正する。

| 状況 | 補正 | 理由 |
|---|---|---|
| イメージが具体的すぎる | **-1段** | 早期にズレを発見できるが、受け入れストレスが大きい。早めに見せる |
| ちょうどいい塩梅 | ±0 | |
| イメージが曖昧 | **+1段** | 完成品に近いものを見せないと判断できない。まず作る |

例: 信頼性=通常（7割）× イメージ具体的 → 閾値 5割。早めに見せて方向確認する。

### 統合: 最終判断はケースバイケース

軸A・B・C は組み合わせ可能だが、**人間とタスクとシチュエーションに応じてケースバイケース**。
表は固定式ではなくガイドライン。エージェントは自分の見立てを Progress.md の日記に書き残す。

### 1.5. 応答性モード（軸B）の挙動詳細

軸A（信頼性）が閾値そのものを決めるのに対し、軸B（応答性）は**閾値を下回った瞬間の振る舞い**を決める。
オーナー指針: 「目の前にいるなら手を止めて一緒に考えてくれるかもしれないし、もう全部おまかせだからやっといてって人もいるし、そもそも寝てるかも」。

| モード | 閾値割れ時の振る舞い | 想定シチュエーション |
|---|---|---|
| `interactive` | 即時質問・ユーザーの応答を待つ | ペアプロ中・横で見ている |
| `relaxed`（デフォルト） | `Questions.md` に質問を置き、別タスクへ移る | 通常の遠隔依頼。ユーザーは別件作業中 |
| `unattended` | `Questions.md` に質問を置き、最有力解釈で**投機続行** | 就寝中・出張中・長時間 `/loop` |

**`unattended` の投機続行ロジック**:

- 閾値を下回った判断は **専用ブランチ**（`speculative/PHASE-XX-投機名`）に commit する
- メインブランチには 9割以上の確度を持つ作業だけが乗る（軸A の閾値とは別建ての安全弁）
- ユーザー復帰時、`Questions.md` と投機ブランチ一覧を読みながら採用 or 破棄を判断できる

これにより「ユーザー不在で全停止」と「不在中に勝手に方向違いを本流マージ」の両方を回避する。

### 1.6. `unattended` の情報伝達フラグ

「ユーザーがどう不在か」をメタファー（就寝中/出張中など）で縛らず、**ユーザーが何を欲しいか**を独立した2フラグで宣言する。
就寝中でも通知が欲しい人もいれば、出張中こそ朝刊式でまとめたい人もいる。シチュエーションでなく欲求で切る。
ユーザーごとに事情が違うため、動作の抽象だけで表現する。

| フラグ | 意味 | デフォルト |
|---|---|---|
| `dashboard_report` | 次セッション冒頭で `.claude/addf/Dashboard.md` を表示するか | `true` |
| `uncertainty_notify` | 閾値割れ・投機分岐・障害時に外部通知（Discord/Slack 等）を送るか | `false` |

直交する2フラグで4パターンを表現:

| dashboard | notify | 振る舞い（典型例） |
|---|---|---|
| ✓ | ✗ | 復帰時にまとめて朝刊で受け取る |
| ✗ | ✓ | 都度通知。ダッシュボードは見ない |
| ✓ | ✓ | リアルタイムで重要事を拾いつつ、まとめも見る |
| ✗ | ✗ | サイレント記録のみ。後で自分でログを掘る |

宣言例:

```yaml
trust: normal
responsiveness: unattended
dashboard_report: true
uncertainty_notify: false
```

ショートカット `/loop --unattended --notify` 等も提供。

#### `dashboard_report: true` の中身

`.claude/addf/Dashboard.md` を自動生成し、次セッション開始時に冒頭表示する:

```markdown
# 前回セッションからの差分まとめ

## 完了
- Plan 0017 サブタスク 3/4 完了
- Plan 0018 サブタスク 1/3 完了

## 投機ブランチ（採否判断待ち）
- `speculative/phase-17-diary-format-A` — 日記フォーマット案A
- `speculative/phase-17-diary-format-B` — 同案B

## 質問（未回答）
- Q1: モックの注入ポイント（Plan 0017 関連）

## 気になった点
- 鮮度低下 knowhow が2件増えました
- 差し戻し1回（テストヘルパーの型不一致）
```

時刻メタファー（「おはようございます」「昨晩」など）は使わない。
夜勤の人や複数タイムゾーンを跨ぐ人の事情と食い違うため、中立に「前回セッションからの差分」とだけ書く。

#### `uncertainty_notify: true` の実装

既存の MCP 統合（Discord 等）を活用する。通知トリガーは以下:

- 閾値割れの判断（質問が `Questions.md` に置かれた）
- 投機ブランチが分岐した
- ビルド/テストが Critical な状態になった
- マイルストーン到達

通知ノイズを抑えるため、重要度しきい値を Plan で宣言可能:

```yaml
notify_threshold: high  # critical | high (default) | medium
```

#### 「本当に止めるべき」条件は2フラグと独立

`uncertainty_notify: false` であっても、本流が継続不能と判断したら**メインの自走は止める**。
止める基準も Plan で宣言できる:

```yaml
critical_when:
  - "ビルド全体が壊れたまま8時間以上経過"
  - "投機ブランチが10本を超えた"
```

これは通知の有無と独立した「自走停止条件」。Dashboard/通知のフラグ設定に関わらず適用される。

### 2. Checkpoint としてのブランチ運用

長大なタスクは中途で「ここから別バージョンを試す」分岐ができるべき。
Checkpoint を Progress.md のテキスト記録だけでなく、**git ブランチそのもの**として扱う:

- サブタスク N まで完了した時点でブランチ `checkpoint/phase-XX-N` を切る（軽量タグでも可）
- 以降の作業で「やっぱり別方針」と判断したら、Checkpoint から `alt/phase-XX-N-別方針` を分岐
- 投機的続行（不在モード）も同じ仕組みに乗る
- 採否は最終的にオーナー判断 or 品質ゲートで決まる

これにより「遺書を残す」発想から「履歴で巻き戻せる作業」へ重心を移す（Plan 0017 と思想を共有）。

### 3. 質問箱 `.claude/addf/Questions.md` の導入

非同期質問チャンネル。エージェントは止まらず、人間が後で答える。

```markdown
# Questions

ループ中のエージェントが「進めない・確信が持てない」と判断した事項を記録する。
人間は気が向いたタイミングで Answer を書く。Answer が書かれたら、エージェントは Plan に反映してから当該質問を「解決済み」に移す。

## 未回答

- 質問: 「視点ずらし」は (a) 別エージェント定義を立てる、(b) 同じエージェントへのプロンプトをロール変更する、どちらを意図していますか？
- 仮の判断: (b) で進めます。違う場合はここに記載してください

### 4. ブートシーケンスへの組み込み

`CLAUDE.md` のブートシーケンス手順 1 に `Questions.md` 読み込みを追加:

```diff
 1. @.claude/addf/Feedback.md を読む — 未対応の改善アクションを確認する
+1.5. @.claude/addf/Questions.md を読む — オーナーから新しい回答がついていれば対応する
 2. @TODO.md を読む — タスクバックログと優先度を把握する
```

オーナーが Answer を書いたら、エージェントは次のセッション開始時にそれを拾って Plan に反映する。

### 5. スキップ可能フラグ

`/addf-dev` 実行時にエージェントが「このタスクは確信度不足でスキップ」と判断した場合の挙動:

- `TODO.md` の該当タスクに `状態: 要確認（質問投下済み）` を立てる
- `Questions.md` に質問を投下
- 次のタスクへ移る（`/loop` の文脈なら自然に次のイテレーションへ）

### 6. worktree モードの宣言

`/addf-dev` 内で worktree を使う場合、Plan に `mode: worktree` を明示する代わりに、`isolation: "worktree"` で起動した時点で自動的に「閾値 50%」とみなすルールを Skill 側に組み込む。
