## 目的

GUI/TUI/モバイルを持つ downstream プロジェクトが参照できるテスト設計ガイドを
`.claude/addf/guides/` に新設し、「AI が毎回画面を見て解釈する」高コスト検証への
構造的な代替（環境依存度による層別＋テキストで返す検証）を文書化する。

## 現状の挙動

- ADDF には GUI テスト機構（addf-gui-test）と環境マトリクス構想（Plan 0029）はあるが、
  「テストをどう層別して設計するか」の一般ガイドラインがない
- Issue #6（2026-05 起票・未回収）が指摘するとおり、画面撮影→AI 解釈の E2E 操作は
  トークン消費が大きく、定常 E2E はコード化して画面を見ずに回せるべき
- Issue #32 が SDIT（TUI プロジェクト）の実運用で同じ問題への解を実証済み:
  - **Layer1: ヘッドレス** — 描画なし・CI 常時実行可能
  - **Layer2: GUI スモーク** — ウィンドウ生成・1フレーム描画のみの軽量チェック
  - **Layer3: GUI/実機操作** — 実操作＋スクリーンショット。権限・実機が必要な最重量層
  - 副次: **テキストのみで返す検証原則**（画像を AI に読ませずツール側で判定し exit code /
    テキストレポートで返す）・**入力チャネルの OS 経由フレームワーク疑い**（IME 等で入力が
    詰まる場合は PTY 直接書き込み等のバイパスを優先候補に）
