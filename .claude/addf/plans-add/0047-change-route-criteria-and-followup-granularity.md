# Plan 0047: 変更ルート判断基準（dev 直行 / 問い合わせ / speculate）とフォローアップ切り出し粒度の再定義

## 実装状況: 未着手

> 出典: 2026-07-07 オーナー起案。「投機実行すべきか開発を進めるべきか、あるいは投機実行を
> 深く掘るべきかの判断基準について」＋「1M コンテキストウィンドウに伴い、フォローアップを
> 別プランに切り出す粒度を、問題のクリティカル度より『プランの主題から外れる、ついでに
> 見つけてしまった何か』に変更するのがよいのでは」

## 関連 Plan

- [Plan 0038: 投機適性の概念化と大改造の窓](0038-speculation-fitness.md) — 投機適性3区分
  （向き/不向き/禁止）は「投機で*できるか*」の除外基準。本 Plan はその手前の
  「オーナー判断を挟む*べきか*」というルート選択の軸を追加する
- [Plan 0041: コンテキスト枯渇とループの壁](0041-context-exhaustion-loop-wall.md) —
  フォローアップ切り出し運用の前提（200k コンテキスト）が 1M 化で変わったことが
  本 Plan 項目2の動機。コンテキスト運用の教義はあちらが正

## 目的

1. タスク完了後のフォローアップや新規変更に対して「main へ直行（addf-dev）/ オーナーに
   問い合わせ / speculative ブランチで作って見せる（addf-speculate 方式）」を選ぶ
   **変更の性質ベースの判断基準**を明文化する
2. フォローアップを別 Plan に切り出す粒度を、200k コンテキスト時代の名残
   （クリティカル度・修正範囲の広さ基準）から、1M コンテキスト前提の
   **「主題からの逸脱」基準**に更新する

## 現状の挙動

- **ルート判断基準が存在しない**: dev と speculate の使い分けは「アイドルかどうか」
  （タスクがあれば dev・なければ speculate）でしか決まっていない。変更の性質
  （バグ修正か・仕様が変わるか・GUI に触れるか）に基づいて「main に取り込んでよいか /
  オーナーの判断を挟むべきか」を決める基準がない
- **フォローアップは別タスクに切り出す運用**: レビュー指摘対応（Progress.md 運用ルール7）は
  クリティカル度で処遇が決まる（Critical/High = フェーズ内 / Medium = 原則修正・先送りは
  独立計画 / Low = Plan 記録）。「バグ分離」ルールは既に関心事ベース（「現在のプランと
  関心事が異なる場合」）だが、切り出しの動機として「広範に修正対象が広がるものを別ターンで
  直したい」という 200k コンテキスト由来の事情が混在している。1M 化により、広範でも主題に
  沿う修正は同一 Plan 内でやりきれるようになった

## 変更内容（項目・フェーズ）

### 項目1: 変更ルート判断基準の明文化

- **対象**: `.claude/addf/guides/speculative-development.md`（新節）、
  `.claude/commands/addf-dev.md`（フォローアップ処理への参照追加）、
  `.claude/commands/addf-speculate.md`（適性判定との接続注記）

変更の性質 → ルートの対応表を定義する:

| 変更の性質 | ルート | 理由 |
|---|---|---|
| バグ修正・セキュリティ修正（挙動が Plan の意図に戻るだけ） | **dev 直行**（main へ取り込む） | 正しさに議論の余地がない |
| バグ修正だが「本来の意図が変わる」or「利用者にとって便利でもあった既存仕様が失われる」 | **オーナー問い合わせ**（トレードオフ提示） | 何が正かはオーナーの価値判断 |
| 機能追加（インターフェース変更なし） | **dev 直行** | 外部契約が不変なら差し戻しコストが低い |
| 機能追加（インターフェース変更あり） | **オーナー問い合わせ** | 外部契約の変更は利用者影響がある |
| GUI 変更を伴う作業 | **speculate 方式**（作って見せる） | 完成イメージは文字より実物 — 軸C `vague` の「まず作る」と同じ理屈 |

- 「オーナー問い合わせ」の提示形態は2択: **(a) 作らずに聞く**（Questions.md /
  AskUserQuestion — トレードオフが文章で伝わる場合）、**(b) 作って見せる**
  （speculative ブランチ + PR — GUI 変更や、見ないと判断できない場合）。
  どちらを選ぶかは軸C（完成イメージ確度）に連動させる: イメージが具体的なら (a)、
  曖昧なら (b)
- **speculate の用途拡張を明記する**: 従来の「アイドル時の直交概念の投機」に加えて、
  「オーナー判断待ち案件の隔離実行方式」としても speculative/ ブランチを使う
  （本流に自動マージされない・承認が起点、という既存の性質をそのまま流用する。
  新しいブランチ規約は作らない）
- 判定は投機適性3区分（Plan 0038）と同じく**特徴ベース**で固定式にしない
  （7割共有原則と整合）。表は「変更の性質」軸、3区分表は「投機で実施できるか」軸 —
  両者の適用順序（ルート判断が先・適性判定は speculate 方式を選んだ後）をガイドに明記する

### 項目2: フォローアップ切り出し粒度の再定義

- **対象**: `.claude/addf/templates/ProgressTemplate.addf.md` の運用ルール7（バグ分離）、
  同期先の `Progress.md`・`ProgressTemplate.md`、
  `.claude/addf/guides/development-process.md`（CLAUDE.md 同期ペアに注意）

切り出し判断の一次軸を**「主題からの逸脱」**に統一する:

- **同一 Plan 内で一緒に進める**（dev でも speculate でも）: Plan の主題に沿うフォローアップ・
  レビュー指摘対応・修正範囲が広くても主題の延長にあるもの。
  「広範だから別ターンに回す」という 200k 由来の切り出し動機を削除する
- **別 Plan に切り出す**: 主題から外れる「ついでに見つけてしまった何か」
  （既存バグの発見・別領域の改善余地など、関心事が現 Plan と異なるもの）
- **クリティカル度の軸は廃止せず役割を変える**: 切り出すか否か（一次軸 = 主題関連性）と、
  切り出した Plan の優先度（二次軸 = クリティカル度）に分離する。
  主題外の Critical/High は「切り出した上で TODO 優先度最上位・次タスクで即着手」とし、
  「フェーズ内先送り禁止」の安全性を粒度変更後も保つ
