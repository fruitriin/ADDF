# worktree 隔離下のエージェント自身が cd で隔離を離脱するリスク（Write/Edit と Bash の非対称性）

> 出典: Plan 0051。`isolation: "worktree"` で起動されたセッションが、状況確認のため共有メインチェックアウトへ
> 絶対パス `cd` を伴う Bash 呼び出しを行い、その後の全 Bash 呼び出しが共有チェックアウト側で実行され
> 続けるヒヤリハットを実地で踏んだことから記録する。

## 発見した知見

### `Write`/`Edit` には隔離保護があるが、`Bash` の `cd` 永続にはない

- worktree 隔離下で共有チェックアウト側のパスに対して `Write`/`Edit` を試みると、ツールが明示的な
  エラー（`This agent is isolated in the worktree ... Edit the worktree copy of this file instead
  of the shared-checkout path.`）で拒否する。これはファイルパス単位のガードであり機能する
- 一方、`Bash` ツールは「作業ディレクトリが呼び出し間で持続する」実装がありうる。共有チェックアウトへ
  絶対パス `cd` を伴うコマンド（例: `cd /path/to/main-checkout && git log ...`）を一度実行すると、
  次に明示的に worktree パスへ `cd` し直すまで、**以降の全ての Bash 呼び出しが共有チェックアウト側の
  working directory で実行され続ける**。この状態で `git commit` を行うと、本来 worktree 側のブランチに
  積むべきコミットが共有チェックアウトの `main`（またはそのとき checkout されているブランチ）に
  直接記録されてしまう
- 両者は「worktree 隔離下のエージェントが誤って共有チェックアウトに影響を与える」という結果は
  似ているが、ガードの有無が非対称という点が本知見の核心

### 実地の実害は限定的だったが、経路自体は再発しうる

- 実測（2026-07-10）では、記録された内容自体（Questions.md への質問投下）が正当な内容だったため
  実害は限定的だった。ただし「本来 worktree 隔離下で main への直接コミットは起こらないはず」という
  前提が崩れた事実は残る。内容次第では意図しない上書き・競合を招きうる
  （`cron-loop-worktree-race.md` が扱う「同一 working tree への複数セッションの再入」とは原因が異なる
  — 本知見は**単一セッション内で自分自身が `cd` により隔離を離脱してしまう**ケース）

## プロジェクトへの適用

- 状態確認のため共有チェックアウト側を覗く必要がある場合:
  - `cd <path> && <command1> && <command2>` のように**1コマンド内で完結させる**（複数の `&&` を
    連ねても、そのコマンド呼び出し限りで作業ディレクトリの変化が閉じる場合はこの形が安全）
  - もしくは、作業ディレクトリ自体を動かさない `git -C <path> <command>` を使う
  - どうしても `cd` だけの呼び出しが必要な場合は、次の呼び出しで**必ず明示的に worktree パスへ
    `cd` し直す**ことを忘れない（本ヒヤリハットは「次で戻すつもり」が抜け落ちて起きた）
- `git commit` / `git push` など変更を確定する操作の直前は、`pwd` と `git rev-parse --show-toplevel`
  を添えて自分が今どの working tree にいるかを確認する習慣が安全網になる
- 本注意は CLAUDE.md「並列実装方針」に汎用的な文言として追記済み（ダウンストリームにも配布される
  ため ADDF 固有の事情を書かない）

## 注意点・制約

- Bash ツールの実装（cwd が呼び出し間で持続するか、毎回リセットされるか）は実行環境・harness の
  バージョンによって異なりうる。本知見は「持続する実装で実際に起きた」事実の記録であり、
  「常に持続する」という前提を全環境に適用してよいとは限らない。ただし**持続しない実装であっても
  1コマンド内で完結させる書き方は安全側に倒れる**ため、習慣として推奨する価値がある
- Bash 経由の `cd`/`git` に対する機械的なガード（PreToolUse hook で worktree 外パスへの操作を
  検知・警告する等）は本 Plan のスコープ外とした。必要性が繰り返し観測されたら別 Plan で検討する
