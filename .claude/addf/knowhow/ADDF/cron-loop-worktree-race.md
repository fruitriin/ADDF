# cron 経由 /loop による /addf-dev 並行実行と working tree 競合

## 発見した知見

- `CronCreate`（`/loop 1h /addf-dev ...`）で自律ループを組むと、cron が発火した新サイクルは
  **同一セッション・同一 working tree** に対して `/addf-dev` を再実行する。CLAUDE.md の
  「並列実装方針」（サブタスクの並列実装は git worktree を使う）は `Agent` ツール経由の委譲を
  想定した記述であり、**cron 発火による `/addf-dev` の逐次/並行再入は対象外**になっている
- 実測（2026-07-10）: 1サイクル目が Plan 0044 を実装中に、バックグラウンドで発火した cron
  ジョブが同じ working tree に対して別サイクルとして Plan 0049（別テーマ）を無コミットのまま
  実装していた。`git status` の外部変更通知（サンドボックスが working tree の変更を検知して
  ツール結果に注記する機構）で発覚した。CronCreate のツール説明は「jobs only fire while the
  REPL is idle (not mid-query)」と明記しているが、長時間ツール呼び出しを連続する中でも
  「idle」とみなされる短い間隙が生じ、そこで発火しうる
- 衝突は `.claude/addf/plans-add/TODO.addf.md` という**両サイクルが触る共有ファイル**の
  **別々の行**で起きた。行が異なったため自然にマージ可能だったが、これは幸運であり保証ではない
  — 同じ行（例: 同じ Plan の状態列）・同じ Plan 番号・同時 `git commit` が起きれば
  非自明な衝突やコミット履歴の破壊につながりうる
- **実測2（2026-07-11・Plan 0052/0053）**: 今回はさらに踏み込んだ衝突が起きた。本セッションが
  リリース準備で `CHANGELOG.md` の `[Unreleased]` セクションに自分のエントリを追記した直後、
  別セッションがその未コミットの追記を「前サイクルの書きかけ」と誤認し、**同じセクション内**に
  自分のエントリ（別 Plan 群の記載漏れ回収）を継ぎ足して Plan 0053 として作業を継続していた。
  行単位では衝突しなかった（互いの挿入位置が異なる行だった）ため実害はなかったが、
  「別々の行」より一段危険な「同一セクション・同一ファイルの重ね書き」が実際に起きた事例
- **実測2の追加知見（同一セクション重ね書きへの対処）**: 自分の追記だけを抽出してコミットする
  手順は以下で再現できた:
  1. 衝突に気づいた時点で、共有ファイルの `git show HEAD:<path>`（衝突前の基準版）と
     現在の working tree 版（両者の変更が混在した版）をそれぞれ別名で退避する
  2. 基準版に対して**自分が行った編集と全く同じ Edit 操作**を再適用し、「自分の変更だけ」の
     版を作る（自分の編集内容は直前のツール呼び出し履歴から正確に再現できる）
  3. 「自分だけ」版と「混在」版を diff し、相手の挿入がどこにあるか正確に特定する
     （逆に、自分の版に相手の内容が一切含まれていないことの確認にもなる）
  4. 「自分だけ」版をファイルに書き戻し、そのファイルだけを `git add`・コミットする
  5. コミット後、退避しておいた「混在」版の内容を working tree に復元する。ただし
     **単純に retrieved した混在版をそのまま貼り戻すのは危険** — 混在版は「自分の版（旧・
     コミット前の状態）＋相手の追記」であり、直前でコミットした「自分だけ」版に対して
     さらに修正（レビュー指摘の反映等）を加えていた場合、混在版に戻すとその修正が消える。
     この時は「相手の追記部分だけ」を diff で切り出し、コミット済みの新しい自分の版に
     手動で再適用するほうが安全（実際に、コミット前に発見した見出し順序バグを先に直して
     から相手の追記を戻す、という手順を踏んだ）
- **実測2の後日談（相手セッションのスタール）**: 上記の停止依頼（TODO.addf.md への注記）を
  残した後、20分・25分の間隔で2回確認したが working tree に進捗が見られなかった。オーナーが
  「向こうは恐らく停止している」と判断し、本セッションが Plan 0053 を引き継いで完走させた。
  相手セッションの正体（cron・`/schedule`〔RemoteTrigger〕・別 VSCode ウィンドウの `/loop` の
  いずれか）は `crontab -l`・`launchd`・`CronList`・`RemoteTrigger action=list` を確認しても
  特定できなかった（いずれも無関係または無効という結果のみ判明。`CronList` は自セッションの
  ジョブしか見えない仕様のため、他セッションの `/loop` が生きていても検出不能）

## プロジェクトへの適用

**衝突を検知したときの対処（実践済み・データ損失なし）**:
1. `git status` で自分が触っていないファイルの変更・新規ファイルを見つけたら、即座に
   `git diff` / `git show HEAD:<path>` で内容を確認する（無視して自分の変更をそのまま
   commit しない — Progress.md 運用ルールの「未コミットの大きな途中状態」を勝手に巻き込まない）
2. 共有ファイルで行レベルの衝突がない場合: `git show HEAD:<path>` で基準版を取得し、
   自分の変更だけを反映した中間ファイルを作ってからそのファイルだけをステージ・コミットする
3. コミット後、相手側の保留中の差分を working tree に復元する（`diff` で完全一致を確認する）
4. 相手が新規作成したファイル・自分が触っていないファイルには一切手を出さない
   （相手の完了処理・品質ゲートに委ねる）

**再発防止の選択肢（今回は「観測継続」で進めたが、次回検討の余地）**:
- cron の interval を長め（例 1h → 2h 以上）にして同時発火の確率を下げる
- `/addf-dev` の冒頭に簡易ロックファイル（例 `.claude/addf/.dev-lock`）を置き、既に別サイクルが
  走っている場合は待機 or スキップする単純な single-flight ガードを追加する（未実装 — 実装するなら
  ロック取得失敗時の扱い〔待つ／今回は idle 扱いにする〕を設計する必要がある）
- cron 発火の `/addf-dev` も `Agent`/worktree 経由に寄せ、CLAUDE.md の並列実装方針の対象に含める
  （ただし `/loop` は「このセッションで会話しながら見る」用途のため、隔離すると自律感が薄れる
  トレードオフがある）

## 注意点・制約

- この競合はサンドボックスの「working tree 外部変更検知」機構に依存して発覚した。この通知が
  ない環境（cron ジョブが別プロセス・別マシンで走る本物の分散環境等）では気づかずに
  `git add -A && git commit` してしまうリスクがある
- 本件は「たまたま別々の行だった」ため実害なしで済んだ。同じ Plan 番号を両サイクルが選ぶ
  （例: 若番優先ロジックが同じ TODO を見て同じ結論を出す）ケースは高確率で起こりうるため、
  cron ループを複数サイクル回す運用では特に注意する
