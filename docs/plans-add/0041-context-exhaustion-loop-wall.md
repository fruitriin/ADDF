# Plan 0041: コンテキスト枯渇によるループ停止の壁の突破

## 実装状況: 未着手

> 出典: オーナー指示（2026-07-06）—「コンテキスト残量が残り少ないときに記録作業に回れるのはいいとして、いよいよ記載することがなくなったときに能動的に compact できなくてループが止まっちゃう壁を突破する計画を考えてほしい」

## 関連 Plan

- [Plan 0023: turn-reminder の関心事分離と能動コンパクション](0023-turn-reminder-context-split.md) — 本 Plan の前提となる残量観測層（context-reminder.py）。本 Plan はその「観測したあと、出口がない」問題を扱う
- [Plan 0017: 代替わり日記](0017-progress-checkpoints.md) — セッション間引き継ぎの既存基盤。フェーズ4（世代交代）の思想的土台
- [Plan 0016: stop-or-go 教義](0016-stop-or-go-doctrine.md) — 「ループ中の突然死」への既存の備え。本 Plan は「突然死」ではなく「自主停止」を扱う

## 目的

/loop 自走中、コンテキスト残量が尽きかけたエージェントが「記録も済んだ、compact する手段もない」の板挟みでループを自主停止する壁を取り除き、コンテキスト枯渇を跨いで自走が継続する状態にする。

## 現状の挙動と壁の構造

### 整備済みの機構（Plan 0017・0023）

1. **観測層**: `context-reminder.py` が実測 180k 超過で「知見記録と日記更新を済ませること。そのまま作業を継続してよい」を注入する
2. **記録層**: 日記（運用ルール 3.5）・`/addf-knowhow` で引き継ぎ情報をファイルに退避する
3. **復帰層**: `post-compact-recovery.sh` が `SessionStart(compact)` でブートシーケンス再実行を注入する。compaction が**起きさえすれば**復帰は機能する（実績あり）

### 壁

- エージェントから compaction を能動的に起こす手段は**存在しない**（2026-07 時点の Claude Code 調査で確定）:
  - `/compact` はユーザー専用スラッシュコマンド。Skill/Bash 経由で自セッションに対して呼ぶ手段なし
  - hooks の `PreCompact` は `decision: "block"`（抑止）のみで、発動させる方向の制御は不可
  - Agent SDK には compaction API（`context_management.edits`、beta `compact-2026-01-12`）があるが、Claude Code CLI には露出していない
- auto-compact は harness が「コンテキスト上限接近時」に自動発動する。つまり **context-reminder 閾値（180k）〜auto-compact 発動点の間にグレーゾーン**があり、ここで記録を書き尽くしたエージェントは:
  - 新規タスクに着手する → コンテキスト超過・性能劣化が怖い
  - compact を起こす → 手段がない
  - の板挟みになり、安全側に倒して **次サイクルを予約せずループを終える**（自主停止）
- さらに 1M variant セッションでは auto-compact 発動点が遠く、実効劣化ゾーン（Opus 系で 250k 前後〜。Fable 系は未計測 — Plan 0023 の残課題）で長時間作業を続けてしまうリスクが別途ある

### 実証済みの突破口: トランスクリプトの「双方向性」（2026-07-06 実験）

トランスクリプト JSONL（`~/.claude/projects/<slug>/<session-id>.jsonl`）とセッションの関係を使い捨てセッションで実験し、以下を確認した:

- **書き込み方向**: セッション → JSONL はリアルタイム追記。ただし実行中のセッションは JSONL を読み直さない（コンテキストはメモリ内の会話履歴から構築される）ため、**実行中の外部編集は現セッションに反映されない**
- **読み込み方向**: `claude --resume <session-id>` 時に JSONL からコンテキストが再構築され、**外部編集が無検証でそのまま反映される**（実験: 「りんご」を sed で「みかん」に置換 → resume 後のモデルは「みかん」を合言葉として回答）
- エントリ行の削除（parentUuid チェーンの断絶）でも resume はエラーにならず会話を継続できた（20行中の assistant 1行削除で検証。大規模削除時のチェーン修復挙動は未検証）
- 構造: 各エントリは `uuid`/`parentUuid` の連結リスト。`tool_use` と `tool_result` のペアリングがあるため、間引きは「エントリ削除」より「`tool_result` の content を短いプレースホルダに置換」（Anthropic API のサーバー側 context editing と同じ発想）が安全

つまり「**セッション終了 → JSONL バックアップ → 選択的に間引く手術 → 同一 session-id で resume**」のサイクルで、能動的・選択的 compaction 相当が原理的に成立する。auto-compact（全体を単一要約に潰す・何が残るか制御不能）と違い、何を残すかをエージェント（または手術スクリプト）が選べ、残した部分は無劣化で保たれる。

### 検討して不採用にした案

- **燃料投下（意図的なコンテキスト消費で auto-compact を誘発する）**: 200k セッションではグレーゾーンが狭くフェーズ2の教義で足りる。1M セッションでは発動点まで数十万トークンを捨てることになり非現実的。どちらのケースでも割に合わない
- **実行中セッションの JSONL を直接編集する**: 上記実験のとおり実行中は読み直されないため効果がない。効くのは resume 経由のみ

## 変更内容（フェーズ）

### フェーズ1: 停止事例の解剖と auto-compact 発動点の実測

- **対象**: 調査のみ（成果物は本 Plan への追記と knowhow）
- 過去の transcript（`~/.claude/projects/` 配下の JSONL）からループ停止事例を特定し、停止の直接原因を分類する:
  - (a) 次サイクル未予約（自主停止） / (b) エラー死 / (c) auto-compact 発動後に文脈を見失った / (d) その他
- auto-compact が実際に発動したセッションの compact 境界前後の `usage` を読み、**発動時トークン数を実測**する（200k セッション / 1M セッションの別も可能なら記録）
- 実測値を `addf-Behavior.toml` の `[context-reminder]` 付近にコメントとして記録し、グレーゾーンの幅を定量化する
- トランスクリプト手術（実装案C）の追加検証: 実サイズのセッションで (a) 大規模な `tool_result` 置換後の resume 成功率、(b) parentUuid チェーン大規模断絶時の挙動、(c) VSCode 拡張セッションと CLI resume の相互運用、を確認する（合言葉レベルの最小実証は 2026-07-06 済み）
- 仮説が外れていた場合（例: 停止原因の主流が自主停止ではない）はフェーズ2以降を実態に合わせて再設計する

### フェーズ2: 「満杯時の出口」教義 — 止まらないことだけがエージェントの仕事

- **対象**: `.claude/addfTools/context-reminder.py` / `.claude/commands/addf-dev.md` / `.claude/templates/ProgressTemplate.md`・`ProgressTemplate.addf.md`（同期ペア lint 対象）
- context-reminder の注入文言に「記録が尽きたあと」の行動指針を追加する:
  - 「記録が済んでいるなら、そのまま次サイクルを続行してよい。compaction を起こすのは harness の仕事であり、ターン境界で自動処理され、復帰フックがブートシーケンスへ導く。エージェントの仕事は**止まらないこと**だけである」
- addf-dev.md のステップ5（ループ継続）に「コンテキスト満杯時の出口」を追記する:
  - コンテキスト満杯を理由にループを止めない。代わりに**省コンテキスト運転**へ切り替える — 1サイクルの作業量を絞る（新規の大タスクに着手しない・サブタスク1個だけ進める・読み込むファイルを最小化する）、日記を書く、次サイクルを通常どおり予約して短いターンで終える
  - auto-compact に巻き込まれて死んだように見えても、日記＋復帰フックで次の代が拾う（それが Plan 0017 で作った家である）ことを明記し、恐れによる自主停止を防ぐ
- ProgressTemplate の運用ルール 3.5 直後に同旨の1行を追加する（テンプレート同期 lint のペア対象確認を忘れない）
- 文言は Plan 0023 の教訓（根拠なき状態断言が早期切り上げを誘発する）を踏襲し、観測事実＋行動指針＋安心文の3点セットを守る

### フェーズ3: 世代交代（generational handoff）— 本当の代替わりをオプトインで

- **対象**: `.claude/addf-Behavior.toml` / `.claude/addfTools/`（新スクリプト）/ `.claude/commands/addf-dev.md` / `docs/guides/development-process.md`
- unattended 自走時（`/addf-mode unattended`）限定・`addf-Behavior.toml` の `[generational-handoff].enable = true` オプトインで有効化する
- 挙動: 実測トークンが handoff 閾値を超え、かつ日記・knowhow の記録が完了したら、エージェントは compaction を待たずに**次世代セッションを自ら起動して正常終了する**:
  - 実装案A: セッション内 Bash から `claude -p "/addf-dev" --permission-mode <現行モード>` を `run_in_background` で起動する
  - 実装案B: 外部ループスクリプト `addf-loop.sh`（while ループで `claude -p "/addf-dev"` を回す）を提供し、/loop（セッション内再帰）をセッション連鎖に置き換える。この場合エージェントは「正常終了する」だけでよく、多重起動リスクが構造的に消える
  - 実装案C（**トランスクリプト手術** — 「実証済みの突破口」参照）: 外部ランナーがセッション終了を検知したら、(1) トランスクリプト JSONL をバックアップし、(2) 手術スクリプトが古い `tool_result` の content をプレースホルダに置換して間引き、(3) `claude --resume <session-id> -p "/addf-dev"` で**同一セッションを圧縮済みコンテキストで再開**する。案Bとの違いは文脈の連続性 — 要約でなく選択的保持なので、残した部分（直近の判断・オーナーとのやり取り）が無劣化で引き継がれる
  - 案B（完全世代交代）と案C（手術 + resume）は同じ外部ランナー基盤に載る変種であり、排他ではない。「日記が引き継ぎに十分なタスクは案B、文脈の生持ちが要る長丁場タスクは案C」の使い分けもありうる。フェーズ1の実測と検証を踏まえて最終判断する
- 案Cの前提リスク: JSONL は非公開フォーマットであり（`version` フィールドあり）、Claude Code のバージョンアップで構造が変わりうる。手術前のバックアップを必須とし、resume 失敗時はバックアップ復元 → 案B（新セッション）へフォールバックする多段構え とする
- コンテキストは引き継がれないが（案A/B。案Cは選択的に引き継ぐ）、それは欠陥ではなく設計である — 日記・Progress.md・Questions.md が引き継ぎチャネルであり、これは Plan 0017 が最初から「本当の代替わり」として想定した形の完全な運用になる
- ガード必須事項: 多重起動防止（ロックファイル）・世代数上限（暴走時のコスト上限）・`uncertainty_notify` フラグとの連携（世代交代を外部通知）

### フェーズ4: 1M variant の実効劣化対策（フェーズ1の実測結果次第）

- **対象**: `addf-Behavior.toml` の `[context-reminder.effective-context]`
- Fable 系の実効コンテキスト目安を実測・記録する（Plan 0023 の残課題の回収）
