# Plan 0027: 通り得ないチェック項目を手順書が持たない仕組み（粗々）

## 実装状況: 未着手

> **粗々の起票**: 設計の方向性と未決事項を出す段階。実装詳細は着手時に詰める。

## 背景

2026-07-02 のレビューで見つかった不具合の上位3つは、すべて「安全網が静かに機能していない」型だった:

1. **lint pair6 の表記ゆれ穴** — 「状態は lint に任せてよい」信用モデルなのに `## 状態:` が黙ってスキップされた
2. **addf-lock の commit ハッシュ** — 手順書に「commit がリリースコミットを指すことを確認」とあるが、
   lock はリリースコミット自身に含まれるため**自己参照で原理的にパス不可能**。2リリース連続で実在しない
   ハッシュが記録され、誰も気づかなかった
3. **addf-experience の死んだ検証** — 検証対象の `@メンション`が1件も存在しないのに検証スキルが生きている

このうち (2) が本 Plan の主対象。**手順書のチェックリストに「構造上パスできない項目」が載っていた**。
手動確認かつ努力目標的な項目だったため、通らないまま「確認した」ことにされ続けた。

これは ADDF の「責めない」文化の裏側にある構造的な罠でもある: **通り得ない項目を持つ手順書は、
それに従った過去のエージェントを静かに嘘つきにする**。手順書が人（エージェント）を責めさせる構造。
本 Plan の狙いは「嘘をつくエージェントを捕まえる」ことではなく「手順書がエージェントに偽の確認を
させない」ことにある。トーンの設計を誤らないこと（Feedback.md オーナーフィードバック参照）。

## 課題の一般化

手順書（`ADDF-Release.addf.md` / `addf-init.md` / `addf-migrate.md` / `ProgressTemplate` の品質ゲート等）の
「〜を確認する／検証する」系ステップには2種類ある:

- **A. 検証可能な事実の主張** — 「lock の commit がリリースコミットを指す」「タグが push されている」など。
  真偽が機械的に判定できる。にもかかわらず裏付けの実行チェックがないと、努力目標に堕ちて theater 化する。
- **B. 人間・エージェントの判断** — 「diff が意図通りか確認」「命名が適切か」など。機械化できない・すべきでない。

問題は **A なのに実行チェックが紐づかず、しかも構造上パス不可能**なケース。lock がまさにそれ。

## 設計の方向性（案・要検討）

### 方向1: 実行可能チェックリスト（有力）

A 型のステップは、真偽を示す**実行可能なアサーション**（exit code で判定できるコマンド）を必ず伴う。
- 例: 「lock の ref がタグとして存在する」→ `git tag -l "$(jq -r .ref lock)" | grep -q .`
- これを品質ゲート／リリース手順で**実際に走らせる**。impossible な項目は初回に FAIL で露見する
  （omission で黙ってパスするのではなく、赤くなる）
- lock ハッシュ問題は、この方式なら「確認する」と書いた瞬間にアサーションが書けず、
  書けたとしても初回実行で必ず落ちるので、設計段階で気づけた

### 方向2: メタ lint（補助）

手順書をスキャンし、「確認する／検証する」動詞を含むステップで**紐づく実行チェック（コードブロック）が
ないもの**を WARNING で列挙する。A/B の区別は完全自動化できないので、
- B（人間判断）は `<!-- human-judgment -->` 等の明示マーカーで除外
- マーカーも実行チェックもない「確認」ステップ = 「信じてね」項目として可視化する

「機械に任せていい」と言った箇所が本当に任せられる状態かの定期点検を、機械化する
（pair6 と同じ「手検査依存の見落とし」を手順書レベルでも塞ぐ）。

### 方向3: 最小対応（下限）

方向1・2 が重ければ、まず既存手順書を人手で棚卸しし、A 型で裏付けのない「確認」項目を洗い出して
実行チェックを付ける／削る。仕組み化せず一回性の掃除で済ませる案。効果は限定的だが低コスト。

- 「責めない」トーンを壊さない出力設計（FAIL 表示が「お前が嘘をついた」ではなく
  「この項目は構造上通らないので手順書側を直そう」と読めること）
