# deprecated モジュールへの保守方針コメント

> 出典: MagiaMagica (コード可視化ツール) Phase 4.0.5〜4.0.9 で確立

段階的移行で「後続フェーズにリメイク/削除予定」となったモジュールに、
**投資優先度を宣言するコメント**を置くパターン。

## 発見した知見

### 問題: AI は「目の前のコードを良くしよう」としてしまう

段階的移行の過渡期間、削除予定の旧実装はリポジトリに残り続ける。
後続セッションの AI がそのコードに触れると、削除予定であることを知らないまま
「このコードを綺麗にしよう」とリファクタ・テスト追加・最適化に投資してしまう —
**捨てるコードへの投資は丸ごと無駄になる**。

計画ファイルや TODO に削除予定が書かれていても、コードを直接読んだセッションには
届かない。**投資判断に効く情報は、判断が起きる場所 (モジュール冒頭) に置く**。

### パターン: 保守方針コメントの3点セット

1. **deprecated マーカーに削除予定フェーズを必ず明記する** — 「いつか消す」ではなく
   「Phase X.Z で消す」。読んだ側が残り寿命を見積もれる
2. **「保守方針 (判定日 YYYY-MM-DD)」節で投資抑制の根拠を書く** — オーナー判定の
   日付を残すことで方針の出典が辿れる。後で方針を覆すときも「いつの判断を覆すのか」が明確になる
3. **優先ルールを明文化する**:
   - バグ修正は受け入れる
   - 軽微な修正・意匠調整は次フェーズのリメイクに織り込む方を優先
   - リファクタ・テスト追加・最適化への投資は避ける

### 実例

```rust
//! **deprecated (Phase X.Y)**: Phase X.Z (新実装への一本化) でリメイクし、本モジュールは削除する。
//!
//! **保守方針 (オーナー判定 YYYY-MM-DD)**: リメイクが終わるまで本モジュールの保守価値は低め。
//! バグ修正・調整の要望が来ても、軽微なら Phase X.Z のリメイクに織り込む方を優先し、
//! ここへの投資 (リファクタ・テスト追加・最適化) は避ける。
```

### 効果

- 後続セッションの AI が「このコードを綺麗にしよう」と無駄な投資をするのを防ぐ
- コードレビューエージェントの指摘 (「テストがない」「重複がある」等) に対して
  「保守方針により先送り」と根拠つきで応答できる
- オーナーの判定日が残るため、方針の出典が辿れる

### 関連: 捨てる前提のファイル隔離 (Temporary Isolation Pattern)

新実装に置き換わる**入口コード**は単独ファイルに隔離し、削除予定を冒頭に明記する。

- 削除予定コードが既存ファイルに混ざっていると、削除時に「どこからどこまでが
  捨てる部分か」の切り分けが要る。最初から単独ファイルなら `git rm` 1つで消える
- 隔離ファイルの冒頭にも上記の3点セットを置く — 「このファイルは丸ごと
  Phase X.Z で消える」が読んだ瞬間に分かる
- 過渡期の取り回し (新旧の分岐・フィーチャーフラグ等) も隔離ファイル側に寄せ、
  本体側のコードを汚さない

## 注意点・制約

- 保守方針コメントは**オーナー判定を伴って**置く。AI の独断で「保守価値は低め」と
  宣言すると、実際は使われ続けるモジュールへのバグ修正まで抑制してしまう
- 削除予定フェーズが流れた (計画変更で延命された) ときは、コメントの方針日付と
  フェーズ番号を更新する。古い日付のまま放置すると出典追跡の価値が逆転する
- バグ修正は常に受け入れる。「保守方針」は品質向上投資の抑制であって、
  壊れたままの放置を正当化するものではない
