# Plan 着手前の実態突合 — 「未着手」表記は信用せず git log で検証する

> 出典: Plan 0015 の完了確認（2026-06-10）。TODO で3ヶ月「未着手」だった計画が実は実装済みだった

## 発見した知見

- TODO の「未着手」と実装の実態はドリフトする。Plan 0015 は 2026-03-21 のコミット `b971d97` で計画ファイル追加と**同時に実装まで完了**していたが、TODO.addf.md は「未着手」のまま3ヶ月放置され、後続セッションのエージェントは皆それを信じていた（Questions.md の Q1 でも「残る未着手は 0015」と扱われた）
- 計画ファイルとその実装が同一コミットに入ると、「Plan を書いた」ことは履歴から分かるが「実装も済んだ」ことが TODO 更新漏れで見えなくなる。完了処理（TODO 反映）が漏れたときの典型パターン
- 突合は安い: `git log --oneline -- <関連ファイル>` や Phase 番号・機能名でのログ検索で1分かからない。二重実装や「実装済みのものを再計画する」無駄を防げる

## プロジェクトへの適用

- Plan に着手する前に、変更対象ファイルの `git log` を確認し、計画内容が既に入っていないか実態と突合する
- 実装済みと判明したら: Plan に実装状況セクションを追記して完了化し、**現状との差分（その後の変更で生じた鮮度ドリフト等）だけを新 Plan に切り出す**。Plan 0015 → 0022 はこのパターン（残差分は example ファイル2件のコピーリスト漏れ等に縮小していた）
- 計画と実装を同一コミットにする場合は、同じコミットで TODO の状態も「完了」にする

## 注意点・制約

- 「実装済みに見えるが計画の一部だけ」のケースがある。完了化する前に Plan の設計項目を1つずつ現物と突合すること（Plan 0015 では実装が計画を超えていた一方、計画後に追加されたファイルへの追従が漏れていた）
